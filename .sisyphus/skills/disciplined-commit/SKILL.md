# Disciplined Commit (Sisyphus)

## 这个技能做什么

把"完成修改后如何提交"标准化为可复用流程。核心原则：

1. **先验证，再提交** — 最贴近行为的验证优先，没通过就不提交。
2. **只暂存相关** — 不把无关改动混进同一个提交。
3. **中文写清楚** — 标题+正文说明背景、修改、验证、后续。

## 何时使用

- 用户要求"提交"、"git commit"、"做一个提交"
- 重要修复/功能/重构完成并通过验证
- 阶段性重构的开始、检查点、完成节点
- 需要把一批变更拆成多个逻辑清晰的提交

## 不适用场景

- 改动未通过最基本验证
- 工作区混有大量无关变更未分离
- 用户明确表示现在不要提交
- 极小临时试验、日志打印、未确认方向的探测修改

## 默认约定

- "重要修改"包括：功能新增、明确 bug 修复、单目标重构、已验证的测试更新。
- 纯文案/纯样式/纯格式化调整不单独视为"重要修改"。
- 每个已验证、可独立说明的变更切片提交一次。
- 阶段性重构默认三类节点提交：`start` / `checkpoint` / `end`，每类必有中文说明。

## 参数

- `主题` — 本次提交核心目标（如"ProjectTree 拖放修复"）
- `阶段编号` — 如 `Phase 22`
- `节点类型` — `start` / `checkpoint` / `end`
- `验证范围` — 如"共享树窄测+架构扫描"

## 工作流程

### 1. 划定提交边界

用一句话概括本次提交。如果说不清，说明改动太散。

对阶段性重构额外确认：
- 是否对应阶段文档中的提交检查点？
- 是否达到该阶段最小结构闭环？
- 是否混入下一阶段工作？

### 2. 先验证，再提交

优先顺序：
1. 与行为最相关的最小验证
2. 被修改模块的窄范围测试
3. 被修改模块的 lint / 类型检查
4. 无更窄验证时，退到 diff 审查

规则：
- 有可执行验证时，不要只看 diff 就提交
- 验证失败 → 先修复 → 再复跑同一验证
- 影响面大时，补更广的回归

### 3. 检查工作区脏变更

明确区分：
- 本次任务改了什么
- 原本就存在的无关修改
- 缓存/构建产物是否误改

### 4. 只暂存相关文件

- 精确文件列表暂存，不用 `git add -A`
- 不顺手提交无关测试、格式化或文档
- 阶段性重构只提交当前阶段相关文件

### 5. 组织中文提交信息

标题：一行中文摘要，可用 `refactor:` / `fix:` / `feat:` 前缀

正文模板：
```text
阶段：
- Phase X / start|checkpoint|end

背景：
- 当前问题或目标

修改：
- 关键变更点

验证：
- 跑过的验证项

后续：
- 未进入本次提交的相邻工作
```

### 6. 执行非交互式提交

- `git status` + `git diff` 确认范围
- `git add <files>` 精确暂存
- `git commit -m "..." -m "..."` 提交

### 7. 提交后汇报

向用户说明：
- 提交是否成功
- 提交摘要
- 主要改动覆盖
- 验证项
- 未提交的剩余变更

## 完成标准

- 能用一句中文准确概括提交目标
- 至少完成一项与改动直接相关的验证
- 暂存区无无关文件
- 提交信息中文，包含阶段/背景/修改/验证/后续
- 未使用用户未授权的破坏性 git 操作

## 模板

```text
refactor: 阶段22-开始 提取 TreeManager 到独立模块

阶段：
- Phase 22 / start

背景：
- ProjectManager(2708行)已超出可维护阈值

修改：
- 从 ProjectManager 中提取 TreeManager 类

验证：
- 项目树展开、折叠、拖放回归测试通过
- lint clean

后续：
- 下一节点将继续提取 DataFileManager
```
