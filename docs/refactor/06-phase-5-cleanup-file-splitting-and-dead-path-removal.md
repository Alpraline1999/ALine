# Phase 5：清理死路径、压平桥接层、拆大文件

## 目标与完成定义

目标：

- 删除隐藏或暂停能力的主链路入口。
- 删除旧桥接层与历史兼容残留。
- 在边界稳定后拆巨型文件。

完成定义：

- 主 UI 链路中不再存在隐藏 AI 面板、隐藏 AI 树入口、旧桥接动作。
- 历史兼容路径不再进入正式运行时。
- 巨型文件按边界拆开，而不是机械切段。

## 进入前提

- `Phase 4` 完成。
- 主边界已经稳定，不再依赖旧运行时路径。

## 本阶段纳入的状态与边界

- 只清理已被新结构替代的旧路径。
- 不在本阶段新增新的业务边界。

## 本阶段禁止改动的区域

- 禁止重新设计目标架构。
- 禁止在拆文件过程中重新引入跨模块私有调用。

## 目标接口/类型/运行时对象

- 本阶段不新增核心接口。
- 只允许对已有边界做文件层面的拆分与清理。

## 实施顺序

1. 清理 AI 隐藏入口、隐藏树节点、隐藏设置入口、隐藏面板路由。
2. 删除旧桥接动作与适配层。
3. 删除已废弃的兼容路径和运行时双轨残留。
4. 按边界拆分巨型文件：
   - `project_tree`
   - `data_page`
   - `chart_page`
   - `digitize_page`
   - `analysis_page`
   - `project_manager` 残留 facade

## 兼容/迁移策略

- 本阶段允许激进清理。
- 只保留对旧项目文件的迁移入口，不保留旧运行时行为。

## 验收标准

- 主 UI 链路不再可达隐藏 AI 能力。
- 旧桥接路径全部移除或隔离出主链路。
- 巨型文件拆分后职责边界与路线图一致。

## 提交检查点

- 检查点 1：隐藏 AI 主链路清理完成。
- 检查点 2：旧桥接与兼容残留清理完成。
- 检查点 3：巨型文件拆分完成。

## 风险与回退办法

风险：

- 清理隐藏路径时误删仍被主链路依赖的逻辑。
- 拆文件时把边界重新打散。

回退办法：

- 如果发现某隐藏路径仍有主链路依赖，先补显式边界，再继续删除。
- 拆文件后若职责不清，优先回退拆分方式，不回退到旧耦合。
