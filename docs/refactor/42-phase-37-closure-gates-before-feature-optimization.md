# Phase 37：功能优化前的结构闭环与质量门槛固化

## 目标与完成定义

目标：

- 在进入功能优化阶段前，对前 37 个重构阶段建立明确的收尾门槛。
- 固化“禁止重新长回 monolith、禁止 UI 直连核心私有实现、禁止重复 command surface”的检查规则。
- 清理剩余规范性、可维护性、一致性和测试结构债务。

完成定义：

- 有一套可重复执行的轻量级审查清单，覆盖模块体量、依赖方向、私有 API 泄漏、重复实现和 UI 一致性。
- 项目内剩余超大 first-party 文件都有明确的后续归属或已完成拆分，不再处于“无人负责”的状态。
- 只有在本阶段验收通过后，才进入功能优化阶段。

## 进入前提

- `Phase 33` 到 `Phase 36` 已完成后续收尾型结构重构。
- 当前代码库已具备稳定的 page shell、service façade 和 command surface 边界。

## 本阶段纳入的范围

- `docs/refactor/README.md`
- 轻量级结构检查脚本或清单
- 目标性静态检查
- `tests/test_ui.py` 等超大测试文件的拆分与组织修正
- UI 风格一致性复核与剩余 token/label 约定补齐

## 本阶段不纳入的范围

- 新业务功能
- 全量回归平台建设
- 新一轮大规模性能优化

## 本阶段禁止事项

- 禁止把“检查阶段”退化成只写结论、不落规则。
- 禁止引入重型全量测试要求，违背本轮窄测约束。
- 禁止为了通过检查，临时增加新的兼容转发层。

## 核心问题清单

- 当前缺少明确的 first-party 文件体量预算与超限处理规则。
- `tests/test_ui.py` 仍是超大测试入口，后续维护成本高。
- 前几轮已暴露出私有 API 泄漏、重复命令面、主题/样式漂移等问题，但尚未全部固化为规则。
- UI 一致性虽然明显改善，但仍需要统一的 settings/card/description label 约定和复核机制。

## 实施顺序

1. 建立结构闭环清单：
   - 大文件预算
   - 私有 API 泄漏扫描
   - 重复 command surface 扫描
   - 页面壳层职责检查
2. 拆分超大测试文件，按页面/模块组织目标性窄测。
3. 做一次 UI 一致性与主题联动复核，固定剩余样式约定。
4. 更新 `docs/refactor/README.md`，标记“可以进入功能优化阶段”的前置条件。

## 验收标准

- 已有可执行的轻量结构检查清单或脚本，后续每轮优化都可复用。
- 不再存在已知但未建账的超大模块或重复实现热点。
- `tests/test_ui.py` 等超大测试入口完成拆分或已明确拆分计划并落在文档中。
- 功能优化阶段的进入条件被正式写入文档。

## 提交检查点

- 检查点 1：结构检查清单/脚本落地。
- 检查点 2：超大测试文件与 UI 一致性收尾完成。
- 检查点 3：阶段验收与“进入功能优化阶段”判定完成。

## 风险与回退

风险：

- 若没有把规则做成可重复执行的检查，后续功能优化阶段会再次引入结构回退。
- 测试文件拆分若不控制范围，容易误触发大规模回归。

回退方式：

- 若某项检查暂时无法自动化，先保留人工 checklist 并在文档中明确执行方式，不强行引入重工具链。
- 若测试拆分影响过大，先做目录与夹具整理，再逐步拆单文件。

## 后续阶段入口

- 只有 `Phase 37` 验收通过后，才进入“功能优化阶段”。
- 功能优化阶段优先关注：
  - 大曲线/大数据量链路性能与交互体验
  - 面向扩展实现的能力增强
  - 算法、分析与批处理体验优化
