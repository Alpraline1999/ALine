# Phase 31：质量门槛回绿、扩展覆盖与警告清理

## 目标与完成定义

目标：

- 把当前“绝大多数通过，但仍存在预存失败/覆盖盲区/升级警告”的状态，推进到真正可持续的质量门槛。
- 修复当前已知的 2 个预存失败测试，并为扩展层建立最小可维护的测试基础设施。
- 清理 `numpy 2.0` 兼容警告及低价值遗留噪声，减少后续重构时的信号污染。

完成定义：

- 当前已知的 2 个预存 backend 失败有明确修复结果，不再被长期搁置为“非本轮引入”。
- 至少为扩展层建立基础测试夹具或 mock runtime，使关键扩展入口不再是 0 覆盖。
- `numpy 2.0` 的 `__array__` copy 弃用警告和明显的空目录/遗留噪声得到处理或明确登记。

## 进入前提

- `Phase 30` 已完成扩展/数字化/AI 包面收口，否则测试基础设施仍会建立在漂移入口之上。
- 当前已知质量债务明确存在：
  - 2 个 backend 失败测试仍为预存问题
  - 扩展层覆盖薄弱
  - `numpy 2.0` 兼容警告仍在

## 本阶段纳入的范围

- `tests/test_backend.py` 中已知失败项
- `tests/` 中与扩展入口直接相关的夹具、mock runtime、窄测
- `numpy 2.0` 兼容告警涉及的模块
- 明显无价值的空目录或遗留噪声

## 本阶段不纳入的范围

- 全量回归策略重写
- 新增大规模 lint 规范迁移
- UI 大文件拆分

## 本阶段禁止事项

- 禁止把预存失败继续作为长期豁免，不形成修复结论。
- 禁止为了追求覆盖率数字而写无行为价值的扩展测试。
- 禁止在无兼容证据的情况下静默忽略 `numpy 2.0` 警告。

## 核心问题清单

- 质量门槛仍不是真正全绿，导致后续阶段每次失败都要先区分“新问题还是老问题”。
- 扩展入口覆盖不足，会削弱任何运行时或包面重构的反馈速度。
- 版本升级警告会掩盖真实回归信号。

## 实施顺序

1. 修复已知 2 个 backend 失败测试，确认是否需要同步调整 mock 契约或环境前提。
2. 建立扩展层基础测试夹具与代表性 smoke matrix。
3. 清理 `numpy 2.0` 兼容警告与明显遗留噪声。
4. 固定新的最小质量门槛说明。

## 验收标准

- 已知 2 个 backend 失败不再存在，或有明确、可复现且被文档记录的环境前提说明。
- 扩展层至少覆盖：
  - 入口加载/调用
  - 参数解析或基础 runtime 契约
  - 1 组处理类或绘图类代表性扩展
- `numpy 2.0` 兼容警告不再出现在相关窄测中。

## 提交检查点

- 检查点 1：backend 预存失败修复完成。
- 检查点 2：扩展测试基础设施与代表性覆盖完成。
- 检查点 3：警告清理、质量门槛说明与阶段验收提交完成。

## 风险与回退

风险：

- 若先补覆盖再修入口，容易让测试建立在即将废弃的表面上。
- 若只压制 warning 而不修兼容点，会制造未来升级债务。

回退方式：

- 若某类扩展难以立即建立真实测试，可先用 mock runtime 固定最小契约，但必须覆盖真实入口。
- 若某条 warning 无法立即消除，先记录到阶段文档并加定向 TODO，不允许直接忽略全部同类警告。

## 延后到后续阶段的问题

- 更大范围的测试分层重写或 CI 体系重构不在本阶段处理。
