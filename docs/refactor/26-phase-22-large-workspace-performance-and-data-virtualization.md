# Phase 22：大工作区性能与数据虚拟化

## 目标与完成定义

目标：

- 在前述结构收口完成后，针对超大曲线、超大数据量和大工作区场景做最后一轮有证据的性能与内存优化。
- 把关注点从“单条曲线 buffer-first”进一步扩展到“整页工作区、整批数据流、整次导入导出”的虚拟化、渐进化和复制预算控制。
- 为未来继续扩展大数据量能力建立可重复的 profiling 样本和性能门槛。

完成定义：

- 大工作区热点有固定 profiling 样本，页面渲染、数据浏览、批处理、导出至少一条链路有可复用的性能基线。
- `DataPage` / 图表渲染 / 导入导出 / 批处理中的至少一组热点实现了虚拟化、渐进渲染或流式执行。
- 大批量曲线和大数据量链路的复制次数、主线程阻塞和峰值内存有明确约束，而不是依赖偶然优化。
- 文档中明确固定规则：
  - 不做“把所有 `list` 一刀切改成 `numpy`”的无证据重写；
  - 只对性能敏感的数值主数据、热点缓存和已测得的瓶颈路径继续数组化或向量化。

## 进入前提

- `Phase 21` 已完成，主要 UI surface 与业务编排边界相对稳定。
- 本阶段必须建立在已有 profiling 证据和热点样本上推进，不允许凭直觉做大面积替换。

## 本阶段纳入的状态与边界

- 纳入：
  - 图表渲染热路径
  - `DataPage` 的大工作区数据浏览与模型刷新
  - `processing/data_engine.py` 的大批量复制热点
  - `ui/dialogs/import_dialog.py`
  - `ui/dialogs/export_flow.py`
  - `core/exporter.py`
  - 直接相关的 profiling / benchmark / fixture 脚本
- 不纳入：
  - 项目文件格式重写
  - GPU 渲染器替换
  - 全仓容器类型强制改为 `numpy`
  - 与性能无关的 UI 改版

## 本阶段禁止改动的区域

- 禁止没有 profiling 证据就做大范围 list-to-numpy 替换。
- 禁止把性能阶段变成另一轮结构性大重构。
- 禁止为了性能跳过必要的正确性窄测与结果一致性验证。
- 禁止用一次性缓存堆积掩盖模型刷新、复制和主线程阻塞问题。

## 实施顺序

1. 先建立大工作区热点样本：
   - 大曲线
   - 多曲线
   - 大树节点/大批量导入
2. 再优化 UI 侧热点：
   - `DataPage` 浏览/刷新虚拟化
   - 图表渐进渲染/抽样/窗口化物化
3. 再优化数据流热点：
   - pipeline 复制预算
   - 导入预览增量解析
   - 导出流式写出
4. 最后固定性能 guardrails：
   - profiling 基线
   - 关键路径窄测
   - 文档化阈值

## 核心问题清单

- 虽然数值主数据已在部分热路径数组化，但大工作区级别的页面刷新、浏览和批量操作仍可能产生明显阻塞。
- `DataPage` 与相关模型在大节点数、大批量曲线场景下仍缺少充分的虚拟化/窗口化策略。
- 图表渲染在超大曲线场景下仍需要进一步控制物化范围和刷新粒度。
- 导入预览、导出执行和批处理链路在大数据量下仍可能产生不必要复制和峰值内存。
- “把所有 list 换成 numpy”看似直接，但会扩大兼容与维护成本，且未必对非热点容器有效。

## 子阶段建议

### 22.1 Profiling Fixtures And Thresholds

目标：

- 建立大工作区/超大曲线基准样本和最小性能阈值。

验收要点：

- 有固定样本可重复触发热点，而不是靠手工临时导入。
- 至少一组关键路径有耗时/内存基线记录。

建议验证：

- 命中的 profiling 脚本
- 必要的 backend / UI 窄测

### 22.2 Workspace Virtualization And Progressive Rendering

目标：

- 优化 `DataPage` 浏览和图表渲染的窗口化、抽样、渐进刷新能力。

验收要点：

- 大工作区浏览与大曲线渲染至少一条路径完成虚拟化或渐进化。
- 页面主线程阻塞与无效刷新次数有明确下降。

建议验证：

- 大样本手工烟测
- 命中的 UI 窄测

### 22.3 Copy Budget And Streaming Execution

目标：

- 优化大批量处理、导入预览与导出的复制预算和流式执行能力。

验收要点：

- 导入、处理或导出至少一条数据流热点不再依赖整批复制。
- 能明确描述关键路径的复制次数、峰值内存或后台执行约束。

建议验证：

- 命中的 backend / import / export 窄测
- 对应 profiling 脚本

## 验收标准

- 本阶段至少有一组性能优化建立在可重复 profiling 证据之上。
- 文档中明确禁止无证据的“全仓 list 改 numpy”式重写。
- UI 浏览、图表渲染、导入导出或 pipeline 批处理中至少一条热点链路实现了实质性性能改进。
- 验证仍然保持窄范围和样本驱动，不扩张为全量回归。

## 提交检查点

- 检查点 1：profiling fixtures 与阈值基线建立完成。
- 检查点 2：workspace virtualization / progressive rendering 首轮优化完成。
- 检查点 3：copy budget / streaming execution 首轮优化完成。
- 检查点 4：阶段验收与性能 guardrails 文档化完成。

## 风险与回退办法

风险：

- 性能优化若没有固定样本，很容易变成主观调参。
- 虚拟化和渐进渲染若边界不清，可能引入显示不完整或交互延迟问题。
- 流式执行和复制预算收口可能误伤现有同步语义。

回退办法：

- 若某项优化没有证据支撑，回退到记录热点和样本，不强推实现。
- 若渐进渲染或流式执行引发功能回归，先保留 profiling 基线，再缩小优化范围。
- 若某段链路尚无明确收益，允许维持当前容器结构，不做无意义数组化替换。

## 延后到后续阶段的问题

- 更大范围的异步作业系统化
- 未来版本级别的项目文件与缓存格式演进
