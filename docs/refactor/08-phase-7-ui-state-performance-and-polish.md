# Phase 7：后置 UI 状态、性能与交互打磨

## 目标与完成定义

目标：

- 处理此前后置的纯 UI 状态。
- 解决长任务体验、性能瓶颈和交互一致性问题。

完成定义：

- 纯 UI 状态拥有统一管理策略。
- 长耗时操作具备明确的反馈、取消或后台执行路径。
- 关键性能路径完成 profiling 与优化。

## 进入前提

- `Phase 6` 完成。
- 核心架构边界和测试门槛已经稳定。

## 本阶段纳入的状态与边界

- splitter 尺寸
- 滚动条位置
- 当前 tab 视觉态
- tooltip 生命周期
- 教学提示临时状态
- 预览工具栏按钮显隐
- 长任务进度、取消、缓存与去抖

## 本阶段禁止改动的区域

- 禁止重新调整核心运行时边界。
- 禁止把纯 UI 状态重新混入业务状态对象。

## 目标接口/类型/运行时对象

- UI 偏好或 UI view-state 模型
- 长任务运行器或后台任务调度接口
- 页面级性能优化策略入口

## 实施顺序

1. 统一纯 UI 状态归属：
   - 留在 view
   - 或进入独立 UI view-state
2. 为长任务建立进度、取消、后台执行机制。
3. 对共享树刷新、扩展扫描、绘图刷新、数字化预览做 profiling。
4. 只在 profiling 结论支持下做缓存、去抖和局部优化。
5. 统一页面空状态、加载状态、错误状态和完成状态。

## 兼容/迁移策略

- 本阶段默认保持功能行为不变。
- 只改体验、性能和纯 UI 状态管理方式。

## 验收标准

- 纯 UI 状态不再污染业务状态对象。
- 长任务不再无反馈阻塞主界面。
- 性能优化有明确前后对比依据。

## 提交检查点

- 检查点 1：纯 UI 状态归属策略落地。
- 检查点 2：长任务机制落地。
- 检查点 3：关键性能路径优化完成。

## 风险与回退办法

风险：

- 在没有 profiling 结论时过早优化。
- 纯 UI 状态抽离过度，反而增加复杂度。

回退办法：

- 若某项优化没有明确收益，回退优化实现，保留测量结论。
- 若某类 UI 状态抽离后使用成本更高，退回到页面 view 内部管理，但不得进入业务状态。
