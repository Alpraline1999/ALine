# Phase 11：大曲线/大数据量热路径收口与内存占用硬化

## 目标与完成定义

目标：

- 在 `Phase 9` 数组主数据和 `Phase 10` 扩展运行时边界稳定后，把大曲线/大数据量主链路进一步收口为 buffer-first 流动。
- 降低图表渲染、处理、分析、数字化和扩展执行中的重复物化、重复复制和主线程阻塞。
- 为未来更激进的列式缓存、临时二进制缓存或后台批处理能力保留扩展点，但不把它们提前混入项目文件格式。

完成定义：

- 图表渲染、处理 pipeline、分析计算、数字化预览/写回、扩展输入准备至少各有一条主链路优先流动 `CurveBuffer` / `SeriesArrayView` / `CurveBatchView`，而不是反复回退到 point-list。
- 高频路径上的 `line -> payload -> DataSeries -> line` 重复物化被收口到少量适配层，页面与服务不再随处重复构造 `x=list(...)` / `y=list(...)`。
- 大曲线/大数据量样本在以下至少两项上具备稳定收益：
  - 减少复制次数
  - 降低主线程阻塞
  - 降低内存峰值
  - 改善大批量批处理吞吐
- 性能收益由固定样本、profiling 输出和窄测共同支撑，而不是凭主观体感判断。

## 进入前提

- `Phase 10` 完成。
- 扩展 runtime 已具备稳定的数组视图输入边界和旧协议收口结果。

## 本阶段纳入的状态与边界

- 纳入：
  - 图表页大曲线渲染、切片、预览、局部刷新和重绘路径
  - 处理页大批量 pipeline、输出批次构建和批量导出准备
  - 分析页大数据量输入准备、结果切换和绘图结果回写
  - 数字化页大点集预览、筛选、写回和导出准备
  - 扩展执行的批量输入准备、惰性物化和结果回写
  - 大曲线/大批量场景下的缓存、所有权和内存生命周期策略
- 不纳入：
  - 项目文件升级为二进制列式存储
  - 对外配置格式改写为数组或二进制协议
  - 全新扩展协议设计
  - 纯 UI 视觉重设计

## 本阶段禁止改动的区域

- 禁止把“所有 `list` 全部替换成 `numpy`”当作目标。
- 禁止让裸 `numpy.ndarray` 无约束扩散到页面控件层、项目文件层和设置持久化层。
- 禁止为了性能回退 `Phase 1` 到 `Phase 10` 已建立的运行时边界。
- 禁止在没有 profiling 证据的情况下，把低频路径或非数值集合强行数组化。

## 目标接口/类型/运行时对象

- `CurveBuffer`
- `SeriesArrayView`
- `CurveBatchView`
- `RenderSlice`
- `CurveMaterializationPolicy`
- `CurveCacheEntry`
- `LargeCurveProfileSample`

## 实施顺序

1. 固定大曲线与大数据量样本：
   - 图表渲染
   - 批量处理
   - 批量分析
   - 数字化大点集
   - 扩展批量输入准备
2. 识别主链路物化点：
   - 标记哪些路径仍在重复构造 point-list、payload 和 `DataSeries`
   - 标记哪些路径必须保留可序列化边界
3. 推进 buffer-first 收口：
   - 让高频计算、批处理和绘图预处理优先使用数组视图
   - 把 point-list / `DataSeries` 物化推迟到 UI 边缘或持久化边缘
4. 收口内存与缓存策略：
   - 统一缓存所有权
   - 明确失效条件
   - 控制大批量场景的峰值内存
5. 固定窄范围验收与 profiling 输出：
   - 保留样本、前后对比和回退策略

## 大规模曲线优化策略

- 优先优化：
  - 大曲线切片与下采样
  - 视口相关的局部物化
  - 批量处理中的输入批视图
  - 分析/数字化中的密集数值向量
  - 结果回写前的惰性构造
- 不优先优化：
  - 树节点集合
  - 配置列表
  - 命令队列
  - 低频 UI 临时状态

如果 profiling 证明需要，可引入更激进的运行时优化形态，例如：

- 临时列式缓存
- 内存映射或磁盘后备缓存
- 大批量后台分块执行

但这些能力必须：

- 只存在于运行时缓存/执行层
- 不改变项目文件格式
- 不让外部扩展被迫直接感知底层存储实现

## 兼容/迁移策略

- 项目尚未发布，可接受内部热路径签名和工作区状态做非兼容优化。
- 但项目文件、设置文件、扩展清单和外部可见配置结构继续保持可序列化 Python 数据边界。
- 如某条链路暂时无法完全 buffer-first，可保留受控物化点，但必须落档说明原因与后续收口计划。

## 验收标准

- 至少三条高频链路不再默认走 point-list 主链路。
- 至少两组大曲线/大数据量样本有可复现的前后对比结果。
- 大批量场景的复制次数、主线程阻塞或内存峰值至少有一项获得明确改善。
- 项目文件兼容性和扩展配置兼容性未被破坏。
- 新增优化没有把裸数组类型扩散成新的 UI / 持久化耦合。

## 提交检查点

- 检查点 1：样本、profiling 输出与热点地图固定。
- 检查点 2：图表或处理主链路完成 buffer-first 收口。
- 检查点 3：分析/数字化/扩展输入至少一条主链路完成收口。
- 检查点 4：缓存、内存策略与窄测闭环完成。

## 风险与回退办法

风险：

- 只替换底层类型，却没有减少主链路物化次数，导致收益不足。
- 数组缓存所有权不清，导致内存泄漏或脏数据复用。
- 为局部性能收益引入过重的运行时复杂度。

回退办法：

- 若某条链路收益不稳定，回退到受控适配层优化，保留 profiling 证据。
- 若缓存策略导致复杂度失控，缩减为只保留高收益样本路径。
- 若某项激进优化触及项目文件或外部兼容边界，立即回退并延期为独立后续议题。
