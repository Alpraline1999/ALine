# Phase 13：代码规范、冗余收口与 UI 风格一致性

## 目标与完成定义

目标：

- 在 `Phase 12` 完成后，对代码规范、兼容层、重复实现和 UI 风格不一致问题做系统性收尾。
- 降低命名漂移、导出面混乱、重复 helper / adapter / presenter 并存带来的长期维护成本。
- 让 UI 的状态呈现、文案风格、页面骨架和共享组件进入稳定可复用状态。

完成定义：

- 仓库中的纯兼容转发层、死路径和重复适配层被清理或拥有明确的退役计划。
- 命名、导入入口、模块导出与 shared helper 的归属更一致，新增代码不再随意扩散工具函数。
- UI 共享 token、共享组件骨架和状态呈现规范已经落地，页面风格趋于一致。
- 窄测中的结构性 warning、明显的重复测试夹具和无主 helper 债务得到收口。

## 进入前提

- `Phase 12` 完成。
- 页面与运行时边界已经稳定，后续规范化工作不会再频繁打断核心结构调整。

## 本阶段纳入的状态与边界

- 纳入：
  - 兼容转发层与死路径清理
  - 重复 helper / adapter / presenter / formatter 收口
  - 模块命名、导出面和导入路径规范化
  - 测试 warning 与重复 fixture 清理
  - UI token、共享组件、空态/错态/加载态/完成态规范
  - 页面标题、按钮、说明文案和状态消息风格统一
- 不纳入：
  - 新业务功能开发
  - 插件市场/联网功能
  - 全新品牌视觉设计
  - 彻底重写现有测试框架

## 本阶段禁止改动的区域

- 禁止为了“看起来更整洁”而夹带行为变更。
- 禁止以规范化名义重新打散 `Phase 12` 刚稳定下来的模块边界。
- 禁止新增新的无主工具模块、匿名共享常量和横向依赖。
- 禁止只清理 import 风格而忽略真正重复实现和兼容债务。

## 目标接口/类型/运行时对象

- `ui/style_tokens.py`
- `ui/presenters/status_presenter.py`
- `ui/presenters/empty_state_presenter.py`
- `core/compat/*` 或等价受控兼容层
- `tests/support/*` 共享测试夹具与工厂
- 仓库级命名/导出约定文档

## 实施顺序

1. 盘点兼容层与重复实现：
   - 纯转发模块
   - 重复 adapter
   - 重复 formatter / presenter
   - 重复测试夹具
2. 固定规范边界：
   - 共享 helper 应落到哪里
   - 兼容层允许存在到什么程度
   - 页面共享 token / 组件 / 状态文案如何组织
3. 清理高噪声债务：
   - 删除无价值转发
   - 合并重复 helper
   - 收口导出面
   - 清理结构性 warning
4. 统一 UI 状态呈现：
   - 空态
   - 加载态
   - 错态
   - 完成态
   - 提示/警告/成功消息
5. 固化 guardrails：
   - 新增架构/规范守卫，防止回潮

## 代码规范与冗余收口策略

- 优先清理：
  - 纯 compatibility forwarder
  - 重复 `line/payload/series/buffer` 适配函数
  - 重复 panel/header/status helper
  - 模块导出与命名不一致
  - 测试中重复的页面/项目构造夹具
- 谨慎清理：
  - 短期内仍承担迁移桥接作用的兼容层
  - 尚无充分替代方案的局部 helper

所有保留的兼容层都必须明确：

- 服务于谁
- 退役条件是什么
- 何时允许删除

## UI 一致性策略

- 统一页面骨架元素：
  - 页面标题区
  - 工具栏/操作区
  - 参数区
  - 状态区
  - 空态/错态提示
- 统一风格 token：
  - 间距
  - 标题层级
  - 常用颜色语义
  - 状态文案语气
- 统一交互反馈：
  - 长任务提示
  - 成功/失败消息
  - 禁用原因说明
  - 批量操作确认文案

## 兼容/迁移策略

- 本阶段优先保持行为稳定，以删除债务和统一约定为主。
- 如某个兼容层仍需短期保留，必须迁入受控兼容目录或在模块头部写清退役说明。
- UI 风格统一应尽量通过共享 token 和共享 presenter 完成，而不是逐页复制样式代码。

## 验收标准

- 纯兼容转发层和重复 helper 显著减少，保留项有明确说明。
- 页面共享状态呈现和常见文案风格具备统一约定，并已在多页落地。
- 窄测中已知的结构性 warning 与重复夹具问题得到实质性改善。
- 新 guardrails 可以阻止明显的 monolith 回潮、随意兼容层新增和共享 helper 扩散。

## 提交检查点

- 检查点 1：兼容层与重复实现清单落地。
- 检查点 2：共享 token / presenter / 文案规范落地。
- 检查点 3：高噪声兼容层、重复 helper 与测试 warning 收口完成。
- 检查点 4：guardrails 与最终收尾验收完成。

## 风险与回退办法

风险：

- 规范化范围过宽，导致阶段拖成“永远收不完”的清理工程。
- UI 一致性抽象过度，压缩页面差异化表达空间。
- 清理重复实现时误删仍有隐性依赖的迁移桥。

回退办法：

- 若某项规范化收益不明显，缩回到 guardrail 和共享约定层，不强推大规模改写。
- 若某个共享组件抽象过重，退回到 token + presenter 层的轻共享。
- 若某条兼容桥仍被依赖，恢复该桥接并立即补充退役说明与替代路径。
