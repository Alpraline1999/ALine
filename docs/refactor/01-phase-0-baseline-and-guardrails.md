# Phase 0：基线与护栏

## 目标与完成定义

目标：

- 为整个重构建立稳定基线、依赖禁令和测试护栏。
- 把“哪些行为必须保持”“哪些结构不允许新增”写成可检查规则。

完成定义：

- 明确当前主流程验证样本。
- 明确架构禁令和依赖扫描规则。
- 明确后续阶段必须持续满足的最小测试门槛。

## 进入前提

- 已阅读 `00-refactor-roadmap.md`。
- 已确认后续阶段会严格遵守本阶段护栏。

## 本阶段纳入的状态与边界

- 只处理“基线定义”和“规则定义”。
- 不改变主运行时结构，不迁移页面业务状态。

## 本阶段禁止改动的区域

- 禁止重写业务逻辑。
- 禁止提前拆页面、拆 `ProjectManager`、拆扩展系统。
- 禁止为了测试方便直接引入新的全局单例。

## 目标接口/类型/运行时对象

- 本阶段不引入新的正式业务接口。
- 只允许引入：
  - 架构扫描规则
  - 依赖禁令清单
  - characterization tests 的样本定义

## 实施顺序

1. 确认当前主流程样本：
   - 打开项目
   - 共享树选择
   - 数据导入
   - 加入绘图
   - 加入处理
   - 运行分析
   - 数字化导出
   - 保存项目
2. 固定后续阶段禁止新增的依赖方式：
   - `ui/**` 不得新增对 `project_manager`、`global_assets`、`extension_registry` 的直接导入
   - `MainWindow` 不得新增对页面私有成员的访问
   - `core/**` 不得新增对 `extensions/**` 的导入
3. 规定测试门槛：
   - 每个阶段至少有一组最贴近改动的窄测
   - 结构性变更需要补架构测试
4. 明确阶段提交流程：
   - 每阶段至少一次开始提交、一次检查点提交、一次完成提交

## 兼容/迁移策略

- `Phase 0` 不做兼容迁移。
- 只确认后续迁移必须通过测试与文档约束驱动，而不是边改边猜。

## 验收标准

- 后续阶段的所有结构性工作都能映射到本阶段定义的护栏。
- 依赖禁令清单明确且可机械检查。
- 主流程样本列表不含歧义。

## 提交检查点

- 检查点 1：基线样本与依赖禁令清单落地。
- 检查点 2：阶段测试门槛与提交流程约定落地。

## 风险与回退办法

风险：

- 基线定义不完整，后续阶段验收失焦。
- 依赖禁令过松，无法真正阻止旧耦合继续扩散。

回退办法：

- 如果后续阶段发现护栏遗漏，先回补本阶段文档，再继续实施。
- 不允许绕过护栏直接推进后续改造。
