# Phase 16：静态质量与可靠性硬化

## 目标与完成定义

目标：

- 对当前仓库里已经系统性暴露的静态质量问题、静默异常问题和高噪声 UI 回调模式做集中硬化。
- 把“现在靠经验避免出错”的部分收口成明确规则，降低后续重构再次引入简单错误的概率。
- 用小步、可验证的方式清掉高噪声 `ruff` 债务，而不是做一次不可控的大扫除。

完成定义：

- `ruff` 高噪声问题按类别分批清掉，至少覆盖：
  - `F401`
  - `F821`
  - `E402`
  - `F841`
  - `E731`
- 静默吞异常路径被分类：
  - 允许静默
  - 需要记录
  - 需要显式反馈
- 高密度 lambda / 闭包回调开始迁移为具名方法或小型 action handler。
- `workspace_state` 代理模式、错误导入/导出面、局部命名漂移得到规范化。

## 进入前提

- `Phase 15` 完成或至少不再阻塞主要页面/widget 结构边界。
- 主要 monolith 的拆分方向已经稳定，静态问题清理不会频繁撞上大规模移动。

## 本阶段纳入的状态与边界

- 纳入：
  - `ui`
  - `core`
  - `app`
  - `processing`
  - `tests`
  - 架构 guardrails / lint 配置 / 文档
- 不纳入：
  - 新功能开发
  - 大规模 UI 结构拆分
  - 算法正确性重写
  - 全量自动修复格式化扫荡

## 本阶段禁止改动的区域

- 禁止直接对整个仓库执行不受控 `ruff --fix`。
- 禁止把行为问题伪装成“只是风格清理”。
- 禁止在没有分类策略的情况下批量删除异常处理。
- 禁止继续新增高密度 property 代理和匿名回调。

## 目标接口/类型/运行时对象

- `ui/callbacks/*` 或等价具名 handler 模块
- `core/errors.py` 或等价异常策略说明
- 仓库级 lint / guardrail 约定
- `PageStateBridge` / `WorkspaceBindings` 规范文档

## 实施顺序

1. 先做依赖与导入类问题：
   - `F821`
   - `E402`
   - `F811`
2. 再做未用导入/变量与导出面清理：
   - `F401`
   - `F841`
3. 再做可靠性相关问题：
   - 静默异常分类
   - 具名回调替换
4. 最后做 proxy / bridge 规范和 guardrails 固化

## 核心问题清单

- 当前 `ruff --statistics` 仍有大量高噪声问题，说明仓库处于“能跑但很容易再漏出低级错误”的状态。
- 多处 `except ...: pass` 直接吞错误，导致运行期问题难以定位。
- UI 层大量 lambda 闭包连接信号，增加调试和重构风险。
- 多处导入/导出面与实际职责不符，容易造成边界漂移。

## 子阶段建议

### 16.1 Import / Export / Name Hygiene

目标：

- 先消灭最容易制造运行时问题的未定义名、非顶层导入和错误重定义。

建议验证：

- 聚焦 `ruff --select F821,E402,F811`
- 命中的 `py_compile`

### 16.2 Noise Reduction

目标：

- 分批清理未用导入、未用变量、错误导出面和明显的局部命名漂移。

建议验证：

- 聚焦 `ruff --select F401,F841,E731`

### 16.3 Exception Policy Hardening

目标：

- 对静默异常做分级，禁止“默认 pass”。

验收要点：

- 每个保留的静默异常都能说明原因。
- 需要记录或反馈的路径有明确去向。

### 16.4 Callback and State Bridge Normalization

目标：

- 让高频 UI 回调与 state bridge 结构更可读、更可调试。

验收要点：

- 热点页面/控件的匿名回调显著减少。
- `workspace_state` 代理扩散趋势被 guardrail 阻断。

## 验收标准

- 关键 `ruff` 类别问题显著下降，并按类别提交而不是混做。
- 静默异常点完成首轮分类，保留项有明确理由。
- UI 匿名回调和 state proxy 模式的增长趋势被扭转。
- 新 guardrails 能阻止简单依赖问题再次回潮。

## 提交检查点

- 检查点 1：依赖/导入类静态问题收口完成。
- 检查点 2：未用导入/变量与导出面噪声收口完成。
- 检查点 3：异常策略与具名回调首轮硬化完成。
- 检查点 4：state bridge 规范与阶段验收完成。

## 风险与回退办法

风险：

- 在清理未用项时误删隐式依赖。
- 在替换 lambda 时打断信号参数绑定。
- 在异常策略硬化时引入过多用户可见噪声。

回退办法：

- 任何静态清理都按小类别和窄测提交，避免一把梭。
- 若某个回调具名化收益低且增加样板，可保留最小闭包形式，但写清边界。

## 延后到后续阶段的问题

- 分析/导入/导出等长流程业务编排重构
- 更深层的共享状态与 presenter 统一
