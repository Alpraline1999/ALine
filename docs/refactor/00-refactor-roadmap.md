# ALine 重构总路线图

## 当前问题总览

ALine 当前最核心的问题不是单个文件过大，而是运行时边界不稳定：

- `MainWindow` 同时承担导航、业务编排、刷新调度和页面内部控制。
- `project_manager`、`global_assets`、`extension_registry` 三类全局状态被多层直接引用，导致状态归属不清。
- 共享项目树既是视图，也是业务入口、动作分发器和全局资源浏览器，耦合过重。
- `ProjectManager` 同时承担项目存取、迁移、树操作、资产 CRUD、备份同步，是系统中心而不是单一职责对象。
- 页面业务状态大量滞留在 `QWidget` 私有属性中，跨页协作只能依赖私有方法和信号网。
- 扩展系统没有形成独立 runtime，页面自行注册扩展，`core` 仍反向依赖 `extensions`。
- 运行时领域模型仍保留 `Dataset` 与 `DataFile` 双轨，兼容路径进入主链路。
- 曲线主数据在热路径里仍大量以 point-list、`x/y` 双列表和临时 payload 来回转换，大数据量场景下对象数量和复制成本过高。

## 目标架构

重构后的系统按 5 个边界组织：

1. `AppShell`
   - 只负责导航、页面装配、命令路由、全局消息订阅。
2. `ProjectSession`
   - 只负责当前项目、树上下文、脏标记、项目级运行时状态。
3. `GlobalAssetCatalog`
   - 只负责跨项目共享的模板、样式、扩展配置。
4. `ExtensionRuntime`
   - 只负责扩展发现、注册、配置适配与调用。
5. `WorkspaceController + WorkspaceState`
   - 每个工作页一套，负责页面业务状态与业务动作。

跨边界补充约束：

- 配置、树节点、项目文件和对外兼容结构继续保持可序列化 Python 数据。
- 性能敏感的数值曲线主数据，允许在运行时收口为数组后端对象。
- 扩展运行时需要基于统一的运行时曲线表示工作，而不是在每个页面、服务和扩展中各自重复转换。

## 阶段划分

- `Phase 0`：建立基线与护栏。
- `Phase 1`：重建 `AppShell` 和运行时边界。
- `Phase 2`：拆 `ProjectManager` 并冻结运行时领域模型。
- `Phase 3`：把页面业务状态从 `QWidget` 中移出。
- `Phase 4`：扩展 runtime 与全局资产子系统化。
- `Phase 5`：清理死路径、移除旧桥接层、拆大文件。
- `Phase 6`：重建质量门槛与测试分层。
- `Phase 7`：处理后置 UI 状态、性能和交互打磨。
- `Phase 8`：大曲线/大数据量性能硬化与 profiling 基线固化。
- `Phase 9`：运行时曲线主数据数组化与热路径表示收口。
- `Phase 10`：扩展运行时与接口契约重构。
- `Phase 11`：大曲线/大数据量热路径收口与内存占用硬化。
- `Phase 12`：超大 UI 页面拆分与页面壳层标准化。
- `Phase 13`：代码规范、冗余收口与 UI 风格一致性。
- `Phase 14`：重复实现清理与架构一致性硬化。
- `Phase 15`：超大页面与共享控件深拆。
- `Phase 16`：静态质量与可靠性硬化。
- `Phase 17`：长流程业务编排与分析工作台收口。
- `Phase 18`：共享处理基础与依赖方向纠偏。
- `Phase 19`：表达式执行与处理扩展契约规范化。
- `Phase 20`：Project Session 服务与命令编排拆分。
- `Phase 21`：UI Monolith 收尾与 Workspace Surface 硬化。
- `Phase 22`：大工作区性能与数据虚拟化。
- `Phase 23`：运行时回归恢复与契约护栏。
- `Phase 24`：多曲线处理性能与数值基础收口。
- `Phase 25`：扩展测试面与模块导出规范化。
- `Phase 26`：项目树与 UI 交互面拆分硬化。

## 阶段依赖

- `Phase 0` 是所有后续阶段的硬前提。
- `Phase 1` 未完成前，不允许实施 `Phase 2` 之后的任何结构拆分。
- `Phase 2` 完成后，才能稳定开展页面状态迁移，因为此时项目状态边界才明确。
- `Phase 3` 完成后，才能安全拆扩展子系统和删除旧兼容路径。
- `Phase 4` 与 `Phase 5` 可局部交叉，但 `Phase 5` 不得先于 `Phase 4` 清理扩展相关死路径。
- `Phase 6` 必须基于前五阶段的结构落位结果编写，不能提前假设目标架构成立。
- `Phase 7` 只处理后置 UI 与性能问题，不承担核心边界调整。
- `Phase 8` 必须建立在 `Phase 7` 的 UI 状态与后台执行路径之上，只处理性能基线、局部优化和 profiling 证据，不回退前述边界。
- `Phase 9` 必须以 `Phase 8` 的热点证据为基础，负责运行时曲线主数据数组化，但不改项目文件格式。
- `Phase 10` 必须建立在 `Phase 9` 的数组主数据契约之上，负责扩展 runtime 与接口协议收口，而不是继续把兼容逻辑堆回单文件。
- `Phase 11` 必须建立在 `Phase 10` 的扩展运行时与数组视图边界之上，负责把大曲线/大数据量主链路进一步收口到 buffer-first 流动，而不是重新引入 point-list 主链路。
- `Phase 12` 必须建立在 `Phase 11` 已固定的热路径和页面协作边界之上，负责拆分超大 UI 页面文件，但不得借拆文件之名重写核心运行时结构。
- `Phase 13` 必须建立在 `Phase 12` 的稳定模块边界之上，负责规范、冗余和 UI 一致性收尾，而不是夹带新的业务功能开发。
- `Phase 14` 必须建立在 `Phase 13` 的规范化结果之上，优先处理已证实存在的重复实现和错误前提，不重新发明扩展或页面边界。
- `Phase 15` 必须建立在 `Phase 14` 的共享基础设施收口之上，继续拆分超大页面和共享 widget，但不得把业务状态散回壳层。
- `Phase 16` 必须建立在 `Phase 15` 的模块边界之上，集中处理静态质量、异常策略和闭包回调等可靠性噪声。
- `Phase 17` 必须建立在 `Phase 16` 的质量收口之上，处理分析、导入导出、处理等长流程业务编排，不改算法目标本身。
- `Phase 18` 必须建立在 `Phase 17` 的流程边界之上，修复共享处理 helper、extension type 与 bootstrap 的基础层问题，而不是继续扩大到表达式或 UI 大改。
- `Phase 19` 必须建立在 `Phase 18` 的正确依赖方向之上，统一表达式执行与扩展参数契约，不引入新的 DSL 或全面行为重写。
- `Phase 20` 必须建立在 `Phase 19` 的底层契约稳定之上，继续拆分 `ProjectManager` 和 `ai.command_layer` 等业务编排中心。
- `Phase 21` 必须建立在 `Phase 20` 的项目级服务和命令边界之上，完成剩余 UI monolith 与 workspace surface 收尾。
- `Phase 22` 必须建立在 `Phase 21` 的稳定结构之上，以 profiling 证据驱动大工作区、大曲线和大批量数据链路的最终性能硬化。
- `Phase 23` 必须建立在 `Phase 22` 的性能与工作区结构之上，优先清除最近一轮 refactor 引入的运行时回归、死代码和过期测试契约。
- `Phase 24` 必须建立在 `Phase 23` 的契约护栏之上，继续处理多曲线与共享数值 primitive 的性能热点，而不是回到无证据的全局数组化。
- `Phase 25` 必须建立在 `Phase 24` 的性能主链路之上，修复扩展测试面、模块导出和低风险命名分裂，不回退 runtime 契约。
- `Phase 26` 必须建立在 `Phase 25` 的测试与模块面稳定结果之上，继续收口项目树和页面交互 surface，但不得重新膨胀主 widget 责任。

## 里程碑

- `M1`：完成 `Phase 0` 和 `Phase 1`
  - 系统具备唯一的壳层、命令模型、事件模型。
- `M2`：完成 `Phase 2` 和 `Phase 3`
  - 项目状态与页面业务状态都拥有明确归属。
- `M3`：完成 `Phase 4` 和 `Phase 5`
  - 扩展系统、全局资产和主业务路径全部脱离历史桥接层。
- `M4`：完成 `Phase 6` 和 `Phase 7`
  - 形成长期可维护的验证体系和后续优化入口。
- `M5`：完成 `Phase 8`
  - 大曲线和大数据量场景具备稳定的性能基线，热点与回归样本固定。
- `M6`：完成 `Phase 9` 和 `Phase 10`
  - 热路径曲线主数据完成数组化收口，扩展运行时与接口契约完成重构并可支撑后续优化。
- `M7`：完成 `Phase 11` 和 `Phase 12`
  - 大曲线/大数据量主链路完成 buffer-first 收口，超大页面文件被拆回可维护模块。
- `M8`：完成 `Phase 13`
  - 代码规范、兼容层、重复实现和 UI 风格一致性进入长期可维护状态。
- `M9`：完成 `Phase 14` 和 `Phase 15`
  - 高收益重复实现收口完成，超大页面与共享 widget 完成第一轮深拆。
- `M10`：完成 `Phase 16` 和 `Phase 17`
  - 静态质量、可靠性和长流程业务编排达到可持续维护水平。
- `M11`：完成 `Phase 18` 和 `Phase 19`
  - 共享处理基础、依赖方向、表达式执行和处理扩展契约完成收口。
- `M12`：完成 `Phase 20` 和 `Phase 21`
  - 项目级服务、AI 命令编排和剩余 UI monolith 完成主要拆分。
- `M13`：完成 `Phase 22`
  - 大工作区、超大曲线与大数据量链路具备稳定的 profiling 基线与虚拟化/流式优化能力。
- `M14`：完成 `Phase 23` 和 `Phase 24`
  - 运行时回归、过期测试契约和多曲线处理热点得到系统化收口。
- `M15`：完成 `Phase 25` 和 `Phase 26`
  - 扩展测试面、模块入口和项目树交互 surface 进入可持续维护状态。

## 风险图

高风险：

- 在 `Phase 1` 前直接拆页面或拆 `ProjectManager`
- 在 `Phase 2` 前移除旧模型
- 在 `Phase 3` 前进行大规模 UI 细节打磨
- 在没有边界约束的情况下把所有 `list` 一刀切改成 `numpy`

中风险：

- 扩展 runtime 清理时遗漏页面侧注册逻辑
- 共享树动作从直接调用改为命令路由时出现行为退化
- 迁移 `Dataset` 到 `DataFile` 时影响旧项目打开
- 在 `Phase 9` 前提前重写扩展协议，导致数组主数据契约和扩展协议脱节

低风险：

- 文档体系搭建
- 提交 skill 升级
- 后置 UI 细节状态整理

## 本轮纳入与后置的状态

本轮纳入的业务运行时状态：

- 处理页：已选输入、当前 pipeline、输出批次、保存目标
- 分析页：已选输入、当前结果、报告模板选择、结果标签上下文
- 绘图页：图表工作集、曲线样式工作集、`FigureState`、扩展实例状态
- 数字化页：当前图片、当前曲线、自动检测预览点、导出目标、校准上下文

后置到 `Phase 7` 的状态：

- splitter 尺寸
- 滚动条位置
- 当前 tab 视觉态
- tooltip 生命周期
- 教学提示临时状态
- 预览工具栏按钮显隐

后置到 `Phase 8` 的优化主题：

- 大曲线渲染抽样、缓存、局部刷新
- 大数据量导入、分析、处理、数字化的后台执行与内存控制
- 性能样本、profiling 输出和热点地图固定

后置到 `Phase 9` 的优化主题：

- 曲线主数据数组化
- `list(point)` / `x/y` / payload 重复转换收口
- 运行时数组缓存和数据视图统一

后置到 `Phase 10` 的优化主题：

- 扩展输入输出的数组原生适配
- 扩展执行 profiling、批处理、去拷贝和错误隔离
- 扩展 loader / registry / invoker / contract 边界重构

后置到 `Phase 11` 的优化主题：

- 大曲线渲染、处理、分析、数字化、扩展执行的 buffer-first 主链路收口
- 渐进渲染、惰性物化、局部切片和大批量结果写回策略
- 大数据量场景的内存峰值、复制次数和主线程阻塞控制

后置到 `Phase 12` 的优化主题：

- `chart_page`、`digitize_page`、`analysis_page`、`process_page`、`settings_page` 的模块化拆分
- 页面壳层、workspace bridge、actions、panels、dialogs 的装配边界统一
- `MainWindow` 与各页的职责回收和调用面缩减

后置到 `Phase 13` 的优化主题：

- 兼容转发层、重复 adapter、重复 helper 和死路径清理
- 命名、模块导出、测试 warning、共享工具接口规范化
- UI token、组件骨架、空态/错态/加载态/完成态的一致性沉淀

后置到 `Phase 23` 的优化主题：

- late-stage refactor 运行时回归扫尾
- 页面 workspace state proxy / command service 回调护栏
- 扩展 runtime 与命令层过期测试契约修复

后置到 `Phase 24` 的优化主题：

- 多曲线处理热路径
- 共享数值 primitive 单源化
- 多输入 pipeline copy budget 与性能 guardrail

后置到 `Phase 25` 的优化主题：

- extension/runtime/command 测试面收口
- `dialogs` / `widgets` 模块导出面规范化
- `HAS_MATPLOTLIB` 等小范围一致性问题

后置到 `Phase 26` 的优化主题：

- `project_tree.py` 高频交互面缩窄

## 路线图约束

- 本文不负责描述逐步实现动作，逐步操作只在阶段文档中定义。
- 任何实施任务必须能映射到某个阶段目标；无法映射的任务不得直接进入开发。
- 阶段顺序变更必须先更新本文，再更新对应阶段文档。
