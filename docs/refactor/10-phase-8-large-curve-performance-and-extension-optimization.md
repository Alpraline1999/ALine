# Phase 8：大曲线/大数据量性能硬化与 profiling 基线固化

## 目标与完成定义

目标：

- 解决大曲线和大数据量场景下的渲染、交互、处理与分析性能问题。
- 为扩展执行路径建立可观测的 profiling 基线和局部优化入口。
- 为后续 `Phase 9` 的主数据数组化和 `Phase 10` 的扩展接口重构提供证据。

完成定义：

- 关键大数据场景具备 profiling 基线、优化策略和前后对比结果。
- 页面渲染、处理、分析、数字化和扩展调用不再默认对全量曲线做无差别同步处理。
- 已明确哪些重复转换和复制热点需要进入 `Phase 9` 收口，哪些扩展接口问题需要进入 `Phase 10` 收口。

## 进入前提

- `Phase 7` 完成。
- 纯 UI 状态、后台执行入口和窄范围验收体系已经稳定。

## 本阶段纳入的状态与边界

- 纳入：
  - 大曲线渲染与视口交互性能
  - 大数据量导入、处理、分析、数字化的后台执行与进度反馈
  - 扩展执行路径的 profiling、缓存、后台入口和局部去拷贝优化
  - 运行时热路径的数据热点识别与优化证据沉淀
- 不纳入：
  - 重新设计 `ProjectSession`、`WorkspaceState`、`GlobalAssetCatalog`、`ExtensionRuntime` 的边界
  - 把运行时曲线主数据全面切换为新的 canonical 数组类型
  - 重构扩展接口协议与扩展运行时模块边界
  - 修改项目文件格式以强制要求历史项目升级到二进制/非 JSON 存储

## 本阶段禁止改动的区域

- 禁止为了性能直接回退已完成的边界拆分。
- 禁止把 `numpy` 或其他数组结构直接扩散成全局持久化格式约束。
- 禁止在没有 profiling 证据的情况下大面积替换实现。
- 禁止让扩展直接依赖页面控件或私有 UI 对象以换取“更快”的调用路径。
- 禁止把 `Phase 9` 的主数据数组化和 `Phase 10` 的接口重构提前混入本阶段。

## 目标接口/类型/运行时对象

- 渲染抽样器 / downsampling 策略接口
- 性能基线记录与 profiling 输出约定
- 热点与重复转换清单
- 后台执行与局部刷新切片约定

## 实施顺序

1. 建立 profiling 基线：
   - 图表页大曲线预览
   - 处理页大批量曲线 pipeline
   - 分析页大数据量计算与结果切换
   - 数字化页大点集预览与写回
   - 扩展执行的输入准备、输出反序列化和配置读取
2. 固定热点地图：
   - 识别哪些路径存在重绘过多、同步阻塞、重复转换、重复拷贝
   - 明确哪些问题归属 `Phase 8`，哪些必须延期到 `Phase 9` / `Phase 10`
3. 只在 profiling 支持下做局部优化：
   - 渲染抽样
   - 局部刷新
   - 结果缓存
   - 批处理和去抖
   - 去拷贝或延迟拷贝
   - 后台执行与主线程解耦
4. 固定性能验收样本与窄测矩阵。

## `numpy` / 数组化策略

本阶段推荐的方向不是“立刻把所有 `list(point)` 曲线格式统一改成 `numpy` 并向全系统扩散”，而是：

- `Phase 8`：
  - 只负责识别哪些路径值得数组化，并在局部热点上验证收益。
- `Phase 9`：
  - 正式负责把运行时曲线主数据收口为数组后端对象。
- `Phase 10`：
  - 正式负责让扩展 runtime 和扩展接口契约与数组主数据对齐。

换言之：

- 不是“所有地方都改成 `numpy`”。
- 而是“先用 profiling 证明哪些路径值得改，再在后续阶段把性能敏感的运行时主数据改成数组表示，并保持存储与兼容边界稳定”。

## 扩展实现优化范围

- 为 plot / processing / analysis / digitize 扩展建立统一输入适配：
  - 标量配置仍走现有配置模型
  - 当前阶段只记录哪些输入准备路径最耗时，正式数组原生契约延后到 `Phase 10`
- 为扩展输出建立标准回写约定：
  - 单曲线输出
  - 多曲线批量输出
  - 元数据/统计结果输出
- 为扩展执行增加：
  - 输入缓存
  - 结果缓存键
  - 批处理候选入口
  - 后台执行入口
  - 错误隔离与超时控制候选点

## 兼容/迁移策略

- 旧扩展默认继续可运行。
- 新增性能适配层时，优先采用“局部优化 + 证据沉淀”：
  - 不在本阶段强行切换扩展签名
  - 不在本阶段要求扩展直接面向数组视图
- 项目文件不因本阶段被迫变更格式。
- 如果未来需要引入更激进的列式/二进制存储，必须作为后续独立阶段，而不是本阶段顺手混入。

## 验收标准

- 至少一组大曲线渲染样本有明确的优化前后对比。
- 至少一组大数据量处理/分析样本有明确的优化前后对比。
- 扩展执行路径的输入准备、结果回写和配置读取热点有 profiling 证据支撑。
- `Phase 9` 和 `Phase 10` 的切入点已经明确落档，而不是停留在口头建议。
- 项目文件兼容性与旧扩展兼容性不因性能优化而被无意破坏。

## 提交检查点

- 检查点 1：profiling 基线与热点地图落地。
- 检查点 2：后台执行/缓存/局部刷新首批优化落地。
- 检查点 3：大曲线渲染/处理/分析关键路径优化完成。
- 检查点 4：性能验收样本、窄测、`Phase 9` / `Phase 10` 输入文档固定。

## 风险与回退办法

风险：

- 把 `numpy` 扩散成全局硬依赖，导致存储边界和兼容边界被污染。
- 只优化局部函数，却没有减少主链路上的复制和重绘次数。
- 过早引入新的扩展接口或新的曲线 canonical 表示，导致阶段边界混乱。

回退办法：

- 若某处局部数组化试点带来边界复杂度上升，退回到 profiling 结论，不把表示变更继续外溢。
- 若某项性能优化没有稳定收益，回退该优化实现，保留 profiling 结论。
- 若扩展执行优化触及接口兼容边界，停止继续外扩，待 `Phase 10` 统一收口。
