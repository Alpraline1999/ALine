# Phase 30：扩展、数字化与 AI 包面合并

## 目标与完成定义

目标：

- 收口当前仍然分散或重复的扩展、数字化和 AI 包面，明确单一拥有者。
- 处理专家审查中仍然合理的结构性问题：薄封装层、双重包面、重复定义和跨层 helper 漂移。
- 为后续测试回绿和大模块拆分提供稳定的单一入口。

完成定义：

- `extension_loader` / `extension_invoker` 这类仅承担薄封装的模块有明确命运：合并、删除或收口为真实 adapter。
- `digitize/` 与 `extensions/digitize/` 的重复类/重复能力有单一实现归属。
- `ai/` 与 `core/ai/` 的双重包面不再同时承担同类职责。
- 若 `processing/smoother.py` 与 `core/line_tools.py` 的共享数值 helper 仍重复，必须明确单一共享层或受控 adapter。

## 进入前提

- `Phase 29` 已完成共享 UI surface 的第二轮拆分，不再把界面噪声和包面合并问题混在一起处理。
- `Phase 24` 的数值 primitive 与 `Phase 25` 的扩展测试/模块面规则仍然有效。

## 本阶段纳入的范围

- `core.extension_*`、`extensions/*`、`digitize/*`、`extensions/digitize/*`
- `ai/*` 与 `core/ai/*`
- 与上述包面直接相关的共享 helper、adapter 和入口模块

## 本阶段不纳入的范围

- 新增扩展能力
- 重新设计数字化交互流程
- AI 配置页或命令层的大型业务改版

## 本阶段禁止事项

- 禁止在未确定单一拥有者前继续复制 helper 或 adapter。
- 禁止把依赖方向问题通过“再加一个中间层”掩盖掉。
- 禁止在未补消费者迁移计划前直接删除有 10+ 消费者的旧入口。

## 核心问题清单

- 扩展入口仍有薄封装层，消费者多时会放大维护成本。
- `digitize` 与 `extensions/digitize` 双重定义使变更边界模糊。
- `ai/` 与 `core/ai/` 同时存在，容易造成配置、模型元数据和命令调用边界再次分裂。
- 少量共享数值 helper 因依赖方向顾虑保留重复实现，需要更稳妥的单源策略。

## 实施顺序

1. 盘点扩展、数字化、AI 三组包面的消费者和单一拥有者候选。
2. 先迁移薄封装层和重复入口，再处理重复实现或 adapter。
3. 为每一类迁移补消费者层窄测，避免只看 import 成功。
4. 清理已废弃的空目录或无消费者转发层。

## 验收标准

- 扩展、数字化、AI 三类包面都能回答“真正入口在哪一层、为何在这一层”。
- 旧转发层要么删除，要么被限制为明确兼容层，不再继续承载新逻辑。
- 重复 helper 不再以“结构性原因”永久悬置，至少形成单源实现或受控 adapter 方案。

## 提交检查点

- 检查点 1：包面盘点与消费者迁移计划完成。
- 检查点 2：扩展/数字化/AI 单一入口收口完成。
- 检查点 3：清理重复层、窄测与阶段验收提交完成。

## 风险与回退

风险：

- 多消费者入口改动若无迁移顺序，容易引入隐性导入回归。
- 若强行把所有重复 helper 拉回同一层，可能再次打破依赖方向。

回退方式：

- 若单步迁移影响面过大，先保留受控兼容层，但必须记录消费者收缩计划。
- 若共享 helper 的真实单层位置暂时无法确定，可先建立专用 adapter 层，而不是继续双向复制。

## 延后到后续阶段的问题

- `ai.command_layer` 的大型编排拆分延后到 `Phase 32`。
- 更大范围的扩展运行时 profiling 与并发优化不在本阶段处理。
