# Phase 6：质量门槛与测试分层

## 目标与完成定义

目标：

- 为新架构建立稳定的质量门槛。
- 把测试从“大而混杂”改成分层验证体系。

完成定义：

- 存在 characterization tests、架构测试、模块级窄测、关键集成测试。
- 每个阶段性回归都有明确最小测试集。

## 进入前提

- `Phase 5` 完成。
- 新架构边界和文件结构已经稳定。

## 本阶段纳入的状态与边界

- 只处理测试体系与验收自动化。
- 不再调整核心运行时边界。

## 本阶段禁止改动的区域

- 禁止以“为了让测试通过”为理由回退已完成的架构边界。
- 禁止重新把集成测试堆回单一大文件。

## 目标接口/类型/运行时对象

- 架构测试规则
- characterization 测试样本
- 页面工作流窄测
- 项目级集成测试分层约定

## 实施顺序

1. 固定 characterization tests。
2. 固定架构测试：
   - 单例导入禁令
   - `MainWindow` 私有调用禁令
   - `core -> extensions` 禁令
3. 按子系统拆分窄测。
4. 提炼关键集成测试。
5. 定义阶段验收测试矩阵。

## 兼容/迁移策略

- 保留历史测试中仍有价值的用例。
- 不保留为了旧结构服务的测试 patch 技巧和大规模 monkeypatch 习惯。

## 验收标准

- 测试结构能直接对应目标架构边界。
- 任一阶段回归都能找到对应的最小测试集。
- 架构测试能阻止旧耦合重新进入代码库。

## 提交检查点

- 检查点 1：characterization tests 固定。
- 检查点 2：架构测试落地。
- 检查点 3：模块级窄测与集成测试分层完成。

## 风险与回退办法

风险：

- 测试分层不清，仍然依赖大规模 patch 才能运行。
- 结构变化后测试覆盖失焦。

回退办法：

- 如果某类测试仍强依赖旧路径，必须先重写该类测试，再继续扩展覆盖。
