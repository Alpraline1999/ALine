# Phase 9：运行时曲线主数据数组化与热路径表示收口

## 目标与完成定义

目标：

- 将性能敏感的曲线主数据从 Python point-list / `list(x), list(y)` 临时结构，收口为统一的数组后端运行时表示。
- 让渲染、处理、分析、数字化的热路径减少 Python 对象拆装、重复复制和重复归一化。
- 在不污染项目文件、Pydantic 模型和配置边界的前提下，为后续扩展运行时提供稳定的数据基座。

完成定义：

- 系统内存在明确的运行时曲线类型，例如 `CurveBuffer`、`SeriesArrayView` 或等价对象。
- 图表渲染、处理 pipeline、分析计算、数字化预览/写回的主链路，优先通过数组后端对象流动。
- `list(point)`、`x/y` 双列表、`series payload` 之间的重复转换被集中到少量适配层，而不是散落在页面、服务和扩展实现中。
- 项目文件与外部兼容边界继续使用可序列化结构，数组对象不直接成为持久化格式。

## 进入前提

- `Phase 8` 完成。
- 已有针对大曲线/大数据量场景的 profiling 证据，能够指出主要热点和重复转换位置。

## 本阶段纳入的状态与边界

- 纳入：
  - 曲线主数据的运行时 canonical 表示
  - 渲染、处理、分析、数字化热路径的数组化迁移
  - 曲线适配层、缓存层和转换策略统一
  - 与 `Phase 10` 扩展运行时对接所需的数据视图契约
- 不纳入：
  - 树节点、命令列表、配置列表、UI 临时状态等非数值集合的 `numpy` 化
  - 项目文件格式升级为二进制列式存储
  - 扩展接口协议的正式重构

## 本阶段禁止改动的区域

- 禁止把“所有 `list` 全改成 `numpy`”作为目标。
- 禁止让裸 `numpy.ndarray` 无约束扩散到整个 UI 层和持久化模型层。
- 禁止为了数组化而回退 `Phase 1` 到 `Phase 7` 已完成的边界拆分。
- 禁止在没有热点证据的情况下，把非数值集合或低频路径强行改写为数组实现。

## 目标接口/类型/运行时对象

- `CurveBuffer`
  - 建议持有 `x: numpy.ndarray` 与 `y: numpy.ndarray`
- `SeriesArrayView`
  - 只读或轻量视图，用于绘图、分析、扩展和局部切片
- `CurveBatchView`
  - 多曲线批量输入视图
- 适配函数
  - `line_to_curve_buffer`
  - `curve_buffer_to_line`
  - `series_payload_to_curve_buffer`
  - `series_payloads_to_curve_batch`

## `numpy` / 数组化策略

本阶段的推荐方向是：

- 可以把“曲线主数据”改为数组后端运行时对象。
- 但不把“所有 `list`”一刀切地改成 `numpy`。

更具体地说：

- 应改成数组后端的内容：
  - 曲线点序列
  - 重采样结果
  - 分析过程中的密集数值向量
  - 大数据量 digitize 预览点集
- 不应改成数组后端的内容：
  - 配置项列表
  - 树节点集合
  - 命令队列
  - 页面控件状态
  - JSON / Pydantic 持久化边界

实现形式上，优先使用“包装过的运行时对象”，而不是在各处直接裸传 `numpy.ndarray`：

- 优先：
  - `CurveBuffer(x_array, y_array)`
  - `SeriesArrayView(curve_buffer)`
- 不优先：
  - 整个系统直接到处传 `ndarray`

曲线内部表示建议优先采用分离的 `x` / `y` 数组，而不是 `shape=(n, 2)` 的点对象列表：

- 更容易与现有绘图、分析和插值逻辑对接
- 更容易做切片、重采样和零拷贝视图
- 更容易控制 dtype 和内存占用

如果某些算法必须使用二维视图，可在局部提供只读拼接视图，而不是把二维点阵列当作全局 canonical 格式。

## 实施顺序

1. 固化热点地图：
   - 标记 `core.line_tools`、`processing.*`、`ui/pages/*`、分析结果回写、digitize 点集写回中的重复转换热点。
2. 落地运行时数组对象与适配层：
   - 提供统一的构造、校验、切片、回写接口。
3. 迁移高收益热路径：
   - 图表渲染
   - 处理 pipeline
   - 分析计算
   - 数字化点集预览与写回
4. 收口缓存策略：
   - 明确数组缓存的生命周期、失效条件和所有权边界。
5. 固定窄范围验收样本：
   - 大曲线
   - 大点集
   - 大批量分析/处理

## 兼容/迁移策略

- 产品尚未发布，因此允许做内部不兼容重构。
- 但不把这种“不兼容”扩散到项目文件和外部配置格式。
- 在 `Phase 9` 内允许：
  - 工作区状态对象调整
  - 服务层输入输出调整
  - 热路径工具函数签名调整
- 在 `Phase 9` 内不允许：
  - 让旧项目因为数组对象而无法加载
  - 让设置/配置持久化直接保存 `numpy` 对象

## 验收标准

- 至少两条高频链路不再默认走 `list(point) -> x/y -> list(point)` 的重复转换。
- 大曲线渲染或分析样本在数组化后有可描述的收益，至少体现为：
  - 减少复制次数
  - 减少主线程阻塞
  - 降低内存占用或对象数量
- 项目文件兼容性未被破坏。
- `Phase 10` 所需的扩展输入数据视图已经稳定到可复用程度。

## 提交检查点

- 检查点 1：运行时曲线对象与适配层落地。
- 检查点 2：图表/处理/分析至少一条主链路迁移完成。
- 检查点 3：digitize 或批量结果写回路径迁移完成。
- 检查点 4：窄测、性能样本与回退策略固定。

## 风险与回退办法

风险：

- 把数组化写成“全系统裸传 `numpy`”，导致边界反而更乱。
- 只把底层换成数组，却保留上层重复转换，收益不足。
- 过早优化低频路径，复杂度大于收益。

回退办法：

- 如果某条链路迁移后收益不稳定，退回到适配层局部数组化。
- 如果某个运行时对象设计过重，保留核心数组缓存与转换入口，缩减公开接口。
- 如果某类页面迁移风险过高，先锁定其它高收益链路，延后该页面切片。
