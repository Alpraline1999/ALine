# Phase 5：大曲线与长耗时任务的异步化和渐进反馈

## 目标与完成定义

目标：

- 把大曲线和长耗时任务的优化，从“内部性能改善”推进到“用户可感知的工作流优化”。
- 避免图表、处理、分析等页面在大输入场景下继续表现为同步阻塞。

完成定义：

- 大曲线绘图、pipeline 执行、分析执行具备明确的后台执行策略。
- 用户可以看到进度、阶段状态，并在合适场景下取消任务。
- 结果可以渐进式返回，而不是所有计算完成后才一次性刷新。

## 当前审查结论

- chart page 已有 `RenderDecimationPolicy` 与 `decimate_xy_for_rendering()`，说明渲染侧已有基础优化。
- process page 当前主要是：
  - `QTimer` 防抖
  - 同步 `_run_pipeline_now()`
  - 简单 `_downsample()`
- analysis page 当前仍是同步运行分析并更新结果视图。
- 这说明当前优化重点还停留在“局部减负”，没有形成统一的异步任务工作流。

## 主要问题

### 1. 同步阻塞仍是默认行为

影响：

- 大输入下界面可能卡顿或无响应。
- 用户不知道当前是否在计算、何时结束、是否失败。

### 2. 进度与取消能力不足

影响：

- 长任务只能等待结束。
- 错误、超时或输入过大时，用户缺少反馈和中断能力。

### 3. 结果回填粒度过粗

影响：

- 图表/处理/分析都更像“黑盒计算”。
- 对大数据调参时，用户交互效率低。

## 推荐优化方向

### A. 建立统一的后台任务壳层

建议建立跨页面共享的任务模型，至少覆盖：

- task id
- task type
- running / completed / failed / cancelled
- progress text
- progress percent
- result payload
- warning / error

收益：

- chart/process/analysis 可以共享一套任务反馈方式。
- 后续扩展若声明支持后台执行，可直接接入。

### B. 设定阈值驱动的异步策略

不是所有操作都要后台化。

建议按阈值切换：

- 小输入：保留同步，减少复杂度
- 大输入或高成本分析：自动后台执行
- 用户也可手动强制后台执行

### C. 分页面优化

chart page：

- 大曲线重绘做分层反馈
- 优先显示 decimated 预览，再补充完整细节

process page：

- pipeline 执行移出主线程
- 参数频繁变更时取消旧任务，只保留最后一次

analysis page：

- 分析执行后台化
- 图表、摘要、表格、报告可分阶段刷新

### D. 取消与过期结果保护

要求：

- 新任务启动后，旧任务结果不得覆盖新状态
- 页面切换或销毁后，后台结果回填必须有 guard
- 参数连续变更时，应丢弃过期结果

## 验收标准

- 大曲线与长耗时操作不再默认卡住 UI。
- 用户能知道：
  - 当前在做什么
  - 是否还在运行
  - 是否可以取消
- 过期任务结果不会回写污染当前页面状态。

## 本阶段不纳入的范围

- 全量并行计算框架
- 跨进程隔离执行
- 重写数值算法本身

## 风险与回退

风险：

- 若后台任务与页面状态没有版本隔离，容易把旧结果写回新页面。
- 若一开始就把所有路径异步化，会显著提高复杂度。

回退方式：

- 先从 chart/process/analysis 的大输入阈值路径做后台执行。
- 先实现进度与取消框架，再逐步把更多长任务接入。
