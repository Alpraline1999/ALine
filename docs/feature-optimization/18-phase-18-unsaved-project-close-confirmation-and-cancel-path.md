# Phase 18：未保存项目关闭确认与取消路径收口

## 目标与完成定义

目标：

- 修复未保存项目在关闭流程中的确认弹窗能力缺口，为关闭确认补齐“取消”路径。
- 统一项目关闭链路中的未保存确认契约，避免不同入口出现不一致的保存/放弃/中止行为。

完成定义：

- 当用户关闭未保存项目时，确认弹窗必须提供“保存 / 不保存 / 取消”三种明确选择。
- 选择“取消”后，关闭流程立即终止，当前项目、当前页面状态与项目树选中状态保持不变。
- 项目根节点右键“关闭项目”、项目树管理相关关闭入口以及主窗口项目关闭路径共享同一确认契约。
- 补齐关闭确认的窄范围测试，只验证未保存关闭路径，不进行大规模回归测试。

## 当前审查结论

该问题本质上不是单一弹窗文案缺失，而是项目关闭交互契约不完整：

- 当前未保存项目关闭确认仅提供“保存 / 不保存”两种结果。
- 关闭行为属于高风险状态切换动作，缺少“取消”意味着用户无法安全退出本次关闭操作。
- 若不同关闭入口各自拼装确认逻辑，后续很容易继续出现某个入口支持取消、另一个入口仍然直接关闭的行为漂移。

用户当前反馈已经明确表明：

- “取消关闭”应是显式第三路径，而不是依赖关闭弹窗窗口或其他隐式交互绕开。
- 新增取消路径后，关闭操作必须可被完全中止，不能留下半关闭、半保存或当前项目切换异常的问题。

## 主要问题

### 1. 未保存项目的关闭确认缺少中止路径

影响：

- 用户在误触“关闭项目”后，只能二选一地保存或放弃，无法维持当前编辑上下文。
- 对包含未落盘更改的项目，这属于明显的高风险交互缺陷。

### 2. 关闭行为契约可能在多个入口间分散实现

影响：

- 即使某一处补上“取消”，若项目树根节点、主窗口关闭逻辑、项目管理相关入口没有共享统一返回值协议，行为仍会不一致。
- 后续继续扩展“关闭项目”命令时，容易再次遗漏取消分支。

### 3. 当前关闭确认缺少“取消后无副作用”的明确验收口径

影响：

- 仅增加一个按钮并不足以说明问题已解决，还必须验证取消后不会触发保存、不会触发关闭、不会改变当前项目。
- 若没有这条验收边界，后续修复可能只停留在 UI 层面。

## 推荐优化方向

### A. 将未保存关闭确认统一收口为三态结果

建议统一定义：

- “保存”：
  - 先执行保存，再继续关闭项目。
- “不保存”：
  - 放弃未保存改动，继续关闭项目。
- “取消”：
  - 立即终止关闭流程，不执行保存，不切换当前项目，不销毁当前项目上下文。

### B. 为项目关闭入口建立共享确认协议

建议至少统一以下入口：

- 项目根节点右键“关闭项目”
- 项目树管理中的关闭入口
- 主窗口内部的项目关闭调用路径

建议要求：

- 所有入口共享同一关闭前确认服务或同一返回值枚举。
- 不允许某个入口单独用布尔值压缩三态结果，否则容易再次丢失“取消”分支。

### C. 明确“取消关闭”后的状态保持规则

至少需要保证：

- 当前项目仍保持打开状态
- 当前项目仍为活动项目
- 当前页面内容、项目树焦点和选择状态不被异常清空
- 不触发保存提示链路之外的附带刷新或资源释放

### D. 补齐窄范围测试守护

建议优先补充：

- 未保存项目关闭弹窗包含三项操作的测试
- 选择“取消”后不关闭项目的行为测试
- 不同关闭入口复用同一确认契约的测试
- 已保存项目关闭路径不受本次修改影响的回归守护测试

## 验收标准

- 关闭未保存项目时，确认弹窗明确提供“保存 / 不保存 / 取消”三项操作。
- 选择“取消”后，项目保持打开，当前项目不切换，未保存内容不丢失。
- 项目根节点、项目树管理入口和主窗口关闭路径的行为一致，不存在某一入口缺少取消分支。
- 相关修改具备窄范围自动化测试，可以稳定守住该交互契约。

## 本阶段不纳入的范围

- 保存系统整体重构
- 项目关闭后的最近项目记录策略调整
- 应用退出流程的全链路重做
- 跨页面统一确认弹窗框架重构

## 风险与回退

风险：

- 若继续用布尔返回值表示“是否继续关闭”，新增“取消”后很容易与“未保存但不保存”混淆。
- 若只修改某个 UI 入口，不统一关闭前确认协议，后续仍会出现入口间行为漂移。
- 若取消分支没有明确阻断后续关闭逻辑，可能表面出现按钮，实际仍继续执行关闭。

回退方式：

- 优先将回退范围限制在项目关闭前确认层，不回退项目保存与项目切换的既有实现。
- 若阶段实施中发现多个入口耦合过深，应先抽出统一三态确认协议，再逐个入口接入，避免局部补丁继续扩散。
