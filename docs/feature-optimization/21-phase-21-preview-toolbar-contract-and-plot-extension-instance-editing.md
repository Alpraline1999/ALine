# Phase 21：预览工具栏交互契约与绘图扩展实例编辑工作流

## 目标与完成定义

目标：

- 将数据管理页、处理页、数据分析页、可视化页中共享的 matplotlib 预览工具栏，继续收口为稳定的一套交互契约，避免 tooltip、缩放模式、按钮状态在不同页面再次漂移。
- 为可视化页右侧扩展面板中的“已加载扩展”列表补齐实例级编辑工作流，支持右键“编辑”与双击打开编辑弹窗，并允许基于当前实例参数执行“刷新加载”覆盖。

完成定义：

- 共享预览工具栏不再依赖页面侧各自重复安装 tooltip/filter，页面只负责业务回调，工具栏自身负责统一的 tooltip 与按钮交互基础行为。
- 可视化页“已加载扩展”列表支持：
  - 右键菜单出现“编辑”
  - 双击列表项直接进入编辑
  - 弹窗中展示当前实例对应扩展的参数编辑界面
  - 底部仅保留“刷新加载”和“取消”两个动作
- “刷新加载”会用编辑后的参数覆盖当前已加载实例，而不是新增一条重复实例。
- 刷新后列表项名称、参数摘要、tooltip、重绘结果、当前选中态保持同步，不破坏实例顺序、目标曲线归属和 authority/sequence 语义。
- 对扩展已失效、目标曲线已失效、参数校验失败、扩展已被卸载等情况，都有明确的提示和安全降级路径。
- 补齐窄范围测试，不做全量 UI 回归。

## 当前审查结论

本轮修复已经确认，预览工具栏虽然表面上已经抽成共享模块，但页面层仍然残留两类问题：

- tooltip 安装契约仍可能在页面层继续分叉：
  - 有的页面没有接 Fluent tooltip
  - 有的页面又重复追加 tooltip filter，造成重复显示
- 这说明“共享工具栏”当前更多只共享按钮和回调壳层，还没有完全共享交互契约和回归防线。

与此同时，可视化页“已加载扩展”列表仍然缺少实例级编辑能力：

- 当前用户如果想修改已加载扩展的参数，只能撤销后重新选择扩展并重新应用。
- 这种路径会打断用户对“当前实例”的心智，尤其在同一扩展允许重复加载、且不同实例绑定不同目标曲线时，成本很高也容易误操作。
- 由于扩展实例还带有：
  - 目标曲线身份
  - curve_display_name 快照
  - authority
  - sequence
  - extension_version
  所以这个需求不是简单地“弹个 JSON 编辑框”即可，需要成 phase 设计。

## 主要问题

### 1. 共享预览工具栏的交互契约仍不完整

影响：

- 只要页面层继续自行安装 tooltip/filter，后续仍可能再次出现“某页不是 Fluent tooltip”“某页重复 tooltip”的漂移。
- 即使功能按钮已经共享，交互一致性仍然不稳定。

### 2. 已加载扩展列表缺少实例级编辑入口

影响：

- 用户无法围绕“当前这条已加载实例”做增量修改。
- 在多实例场景下，撤销再重建会打断序列语义，也不利于比较前后结果。

### 3. 刷新加载不是简单的再次 apply

影响：

- 若直接重新 apply，很容易变成新增一条实例，而不是覆盖原实例。
- 覆盖时还必须保留原实例的：
  - 列表顺序
  - 目标曲线绑定
  - authority / sequence 语义
  - 当前选中状态

### 4. 编辑器不能只依赖临时文本框

影响：

- 扩展参数已经存在字段定义、说明、校验与默认值体系。
- 若编辑弹窗不复用现有扩展参数编辑组件，就会出现字段渲染不一致、校验链路分裂、维护成本上升的问题。

### 5. 运行时失效场景需要单独治理

影响：

- 扩展可能在“重载扩展”后被卸载或版本变化。
- 目标曲线可能已不存在。
- 如果这些情况没有清晰反馈，刷新加载会制造新的运行时不确定性。

## 推荐优化方向

### A. 把预览工具栏继续收口为“共享交互组件”

建议要求：

- 工具栏自身负责 Fluent tooltip 安装、按钮基础状态和缩放模式同步契约。
- 页面只传入业务回调，不再各自拼装 tooltip/filter。
- 对已有页面的接线，优先删掉页面层重复的 tooltip 逻辑，而不是继续增加局部补丁。

### B. 为已加载扩展实例引入“编辑实例”对话框

建议工作流：

- 右键已加载扩展项，出现“编辑”。
- 双击同一项，直接打开编辑弹窗。
- 弹窗标题明确标识当前扩展名称、目标曲线和实例摘要。
- 弹窗主体复用现有扩展参数编辑组件，而不是新造一套轻量表单。

### C. 明确“刷新加载”的覆盖语义

建议契约：

- “刷新加载”覆盖当前实例，不新增实例。
- 覆盖后保留原实例 `id` 与排序位置。
- 若当前实现里 `sequence` 参与最终叠加顺序，应明确是：
  - 保留原 sequence
  - 还是更新 sequence 但仍保持原列表位置
- 上述语义必须统一，不能边实现边漂移。

### D. 补齐实例编辑过程中的失效防护

建议至少覆盖：

- 扩展类型已不存在时，禁止刷新加载并提示需要移除或重新选择扩展。
- 目标曲线失效时，允许用户明确看到“原目标已失效”，并定义刷新后是失败、忽略还是回退到“全部可见曲线”。
- 参数校验失败时，不修改原实例。
- 扩展版本变化时，应提示兼容风险。

### E. 把列表刷新与重绘同步视为同一个事务

建议要求：

- 实例编辑成功后，列表文本、tooltip、选中项、扩展面板当前类型、图表重绘应一次性同步完成。
- 不允许出现“图已经刷新，但列表还是旧参数”或“列表更新了，但实际图像没变”的分裂状态。

## 验收标准

- 共享预览工具栏在相关页面不再出现非 Fluent tooltip 或重复 tooltip。
- 可视化页“已加载扩展”列表支持右键“编辑”和双击打开编辑弹窗。
- 编辑弹窗支持“刷新加载 / 取消”两条清晰路径。
- 刷新加载后，当前实例被覆盖更新，而不是新增重复项。
- 多实例、失效曲线、扩展重载、参数校验失败等场景具备清晰反馈与安全行为。
- 相关改动有窄范围测试守护。

## 本阶段不纳入的范围

- 绘图扩展协议的全面重写
- 所有扩展类型统一迁移到实例对象模型
- 批量编辑多个已加载扩展实例
- 跨页面统一的扩展实例历史/撤销系统

## 风险与回退

风险：

- 如果“刷新加载”实现成“删除旧实例再新增新实例”，很容易破坏顺序、选中态和目标曲线语义。
- 如果不复用现有扩展参数编辑组件，会很快形成第二套配置 UI。
- 如果只补 UI 入口，不先定义覆盖语义和失效策略，后续会出现更多隐藏行为差异。

回退方式：

- 优先将回退范围限制在实例编辑入口与刷新加载逻辑，不动现有扩展注册与渲染链路。
- 若本阶段中发现实例级覆盖语义无法一次收口，优先先保住“可打开、可校验、可安全取消”的编辑弹窗，再推进覆盖刷新逻辑。
