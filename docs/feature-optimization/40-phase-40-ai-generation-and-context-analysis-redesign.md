# Phase 40：AI 生成与上下文分析模块重设计

## 目标与完成定义

**目标**：重做 AI 模块的底层架构，使其主要面向两类任务：

- 根据用户需求生成扩展脚本和各类模板
- 对项目数据、分析结果、分析报告进行上下文分析

**完成定义**：
- 明确 AI 模块的任务边界、上下文模型和执行入口
- 明确哪些内容可以被直接引用为 AI 上下文
- 形成不依赖当前 UI 上线状态的底层设计方案

## 能力目标

### A. 生成类能力

- 生成处理扩展 / 分析扩展 / 绘图扩展 / 数字化扩展脚本
- 生成绘图样式模板、曲线样式模板、pipeline 模板、扩展配置、分析报告模板

### B. 分析类能力

- 对数据节点进行数据分析
- 对分析结果进行总结、比较、解释
- 对分析报告进行结构化审阅和补充

## 可直接引用为上下文的内容

- 数据节点
- 分析结果
- 分析报告
- 扩展接口文档
- 扩展脚本功能与参数说明
- 各类模板与扩展配置

## 关键设计问题

### 1. 上下文模型

- 引用是传摘要、快照，还是原始对象
- 上下文是否统一抽象为 `ContextReference`

### 2. 任务模型

- 生成脚本/模板与分析任务是否共用同一调度入口
- 是否区分“草稿生成”“结构化修改”“只读分析”

### 3. 安全与边界

- 生成结果如何进入项目树或全局资源
- AI 是否允许直接写入扩展脚本目录
- AI 是否只产出草稿，由用户显式确认保存

### 4. 与 provenance 的关系

- AI 分析应能消费 Phase 39 的 lineage 结构
- AI 生成的模板/扩展也应留下来源记录

## 推荐实施顺序

1. 先定义上下文引用协议
2. 再定义生成任务与分析任务的统一执行模型
3. 再决定 settings / 独立页面 / 隐藏工作台的 UI 挂载方式
4. 最后才考虑重新上线用户入口

## 本阶段暂不拆任务

- 该模块仍处于底层重设计阶段，先做设计，不做原子任务分解
