# Phase 6：数据导入、预览与来源链路优化

## 目标与完成定义

目标：

- 把项目树导入、DataPage 导入、源文件预览和后续数据来源追踪统一成更顺畅的“数据进入系统”工作流。

完成定义：

- 数据导入入口口径更一致。
- 用户在导入前能更明确地预检和预览。
- 导入后的 dataset / series 与 source file / import 参数之间的来源关系更容易追踪。

## 当前审查结论

- 项目树右键菜单已支持：
  - 导入源文件
  - 导入数据文件
  - 导入数字化图片
- DataPage 也有自己的 source preview、pending import、导入和新建数据集逻辑。
- 这说明“功能已经存在”，但入口、预览、导入摘要和来源展示仍是多个局部流程的拼接。

## 主要问题

### 1. 导入入口分散

影响：

- 用户在项目树和 DataPage 之间切换时，容易遇到不同的确认方式、目标绑定和反馈口径。

### 2. 预览与导入参数之间的联系不够强

影响：

- 用户可能在 preview 中试过参数，但导入时又需要重新确认或重新理解当前状态。

### 3. 来源追踪不足

影响：

- 导入后的数据列虽然已经进入项目，但“来自哪个源文件、用了什么解析方式、导入到哪个数据文件”仍不够直观。
- 后续进入 process / analysis 时，追溯来源成本高。

## 推荐优化方向

### A. 统一导入意图模型

建议把导入动作统一成更清晰的几类意图：

- 导入为 source file
- 导入为 dataset/data file
- 导入为 digitize image

每类意图都统一包含：

- 来源路径
- 目标父级
- 命名策略
- 是否覆盖/追加
- 预览参数

### B. 强化导入前预检

建议新增或加强：

- 文件类型识别摘要
- 表头/列预览
- 工作表/分隔符/跳行等解析配置预检
- 重名/冲突目标提示
- 批量导入预估结果摘要

### C. 统一导入后摘要反馈

建议导入完成后明确反馈：

- 新建了哪些 data file / series
- 跳过了哪些文件
- 使用了哪些关键导入参数
- 是否有警告或自动修正

### D. 建立更可见的 provenance 信息

建议在数据对象的可见层面提供来源摘要，例如：

- 源文件名
- 导入时间
- 解析配置摘要
- 后续处理链入口链接

重点不是一开始就做很重的 lineage 系统，而是先把“来源信息可见”做好。

## 验收标准

- 项目树导入与 DataPage 导入口径更一致。
- 用户能在导入前更容易判断“导入后会得到什么”。
- 导入后的数据对象可追溯到来源文件和关键导入参数。

## 本阶段不纳入的范围

- 全新数据库式 lineage 系统
- 云端导入
- 大规模文件格式插件化

## 风险与回退

风险：

- 若一开始就把 provenance 设计得过重，会拖慢实现并扩大持久化改动面。
- 若导入摘要和真实结果不一致，反而会增加困惑。

回退方式：

- 先做轻量来源摘要和导入后反馈。
- 先统一入口口径和预检流程，再逐步增强 lineage 深度。
