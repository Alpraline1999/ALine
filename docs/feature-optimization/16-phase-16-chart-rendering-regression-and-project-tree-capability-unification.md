# Phase 16：可视化渲染回归与项目树节点能力统一

## 目标与完成定义

目标：

- 修复可视化页在树节点激活后触发的绘图运行时崩溃，恢复基础绘图工作流。
- 收口项目树“根分组文件夹 / 普通子文件夹 / 项目根节点 / 可批量操作节点”的能力契约，避免菜单、拖放、节点管理和管理对话框继续各自漂移。
- 让 `ProjectTreeWidget`、项目树管理对话框和 DataPage 节点管理面对同一类节点给出一致的名称、动作和限制。

完成定义：

- 可视化页激活数据文件/数据列后不再因 `decimate_xy_for_rendering()` 参数不匹配而崩溃。
- 普通子文件夹支持重命名、删除、移动；根分组文件夹仍保持受保护。
- DataPage 节点管理面对普通子文件夹显示真实名称，不再统一显示为“源文件”“数据集”等父分组名称。
- 同类节点的批量删除、批量移动和批量拖动移动拥有统一规则，并在项目树与项目树管理中一致生效。
- `source_file` 在源文件集合内支持节点移动，同时保留拖到数据集/数字化集合时的导入语义。
- “清理空文件夹”根级入口恢复可用，并补足 `root_id=None` 路径守护。
- 项目根节点右键菜单新增“保存项目 / 另存项目 / 关闭项目”，并复用统一项目命令。

## 当前审查结论

本轮问题不是单一偶发回归，而是两类独立问题叠加：

- 一类是可视化页渲染抽样接口漂移，已经进入运行时主链路。
- 另一类是项目树节点能力矩阵没有形成统一契约，导致右键菜单、拖放、批量动作、节点管理和项目命令各自实现、各自失真。

已确认现状：

- `core/rendering.py`
  - `decimate_xy_for_rendering()` 现签名为 `(xs, ys, policy=None)`，返回 `(xs_dec, ys_dec, indices)`。
- `ui/pages/chart_page.py`
  - `_redraw_now()` 已改为传入 `_CHART_RENDER_DECIMATION_POLICY` 并接收 3 个返回值。
  - 但 `_decimated_redraw()` 仍调用旧签名 `decimate_xy_for_rendering(list(xs), list(ys), max_points=200)`，因此在大数据量预览路径上直接触发 `TypeError`。
- `ui/widgets/project_tree.py`
  - `_create_child_folder()` 会把普通子文件夹的 `group_type` 写成所属集合，例如 `datasets`、`source_files`。
  - `_is_protected_folder()` 仅按 `group_type` 判断受保护，未区分根分组与普通子文件夹，导致所有子文件夹被误判为系统文件夹。
  - `_move_target_choices()` 没有 `folder` 分支。
  - `_batch_action_payloads()` 只接受 `series` 和 `curve`，其余节点天然失去批量删除/移动/拖动能力。
- `ui/widgets/project_tree_menu_commands.py`
  - `folder` 右键菜单只有“重命名 / 删除 / 清理空子文件夹”，没有“移动到...”。
  - `project` 节点菜单直接退化成树作用域动作，没有项目级保存/另存/关闭入口。
- `ui/widgets/project_tree_drag_drop.py`
  - `source_file` 拖动路径被直接分流到 `perform_source_file_drop_action()`，只处理“导入到数据集/数字化”，不会处理“在源文件集合内移动节点”。
  - `perform_batch_drop_move()` 依赖 `_batch_action_payloads()`，因此当前批量拖动能力也被限制在 `series` / `curve`。
- `ui/pages/data_page.py`
  - `_current_node_name()` 对所有 `folder` 一律优先显示 `_folder_group_label(group_type)`，这会把普通子文件夹名称错误显示为“数据集”“源文件”等集合名。
  - DataPage 自身的 `_is_protected_folder_node()` 已区分根分组与普通子文件夹，和 `ProjectTreeWidget._is_protected_folder()` 的规则并不一致。
- `core/project_tree_service.py`
  - `remove_empty_folders(root_id=None)` 的 service 级行为经窄范围探针可正常移除整棵树中的空子文件夹，因此“清理空文件夹无效”更可能是 UI 入口、项目上下文或缺少守护导致的问题，而不是底层删除算法本身失效。

## 主要问题

### 1. 可视化页仍残留旧抽样 API 调用

影响：

- 只要进入大点数预览路径，绘图就会在运行时崩溃。
- 当前 `chart_page.py` 内部已经同时存在“新接口调用”和“旧接口调用”，后续继续改渲染逻辑时会再次埋下同类回归。

### 2. 根分组语义和普通子文件夹能力被混为一谈

影响：

- 普通子文件夹继承了系统分组的“受保护”语义，导致右键菜单、管理对话框和移动能力异常。
- 这与 phase14 已经建立的“根分组 icon / 普通子文件夹 icon”方向冲突，说明当前项目树仍缺少统一的 folder semantic helper。

### 3. 项目树批量操作模型只覆盖虚拟叶节点

影响：

- 除 `series` / `curve` 外，其它节点都没有批量删除、批量移动和批量拖动移动能力。
- 批量能力的限制并不是业务规则，而是 `_batch_action_payloads()` 的实现残留。

### 4. `source_file` 的“导入语义”和“节点移动语义”没有分开

影响：

- 拖动源文件到数据集/数字化集合时应视为导入，这是合理的。
- 但拖动源文件到源文件文件夹时也走导入分支，导致源文件节点在其本集合内无法移动。

### 5. 根级清理和项目根命令缺少统一入口契约

影响：

- “清理空子文件夹”是 folder-scoped 动作，当前可用。
- “清理空文件夹”是 project/tree-scoped 动作，但没有针对 `root_id=None` 的 UI 路径守护，用户面已经出现失效反馈。
- 项目根节点目前也没有承接保存/另存/关闭等项目级命令，导致项目根右键菜单不完整。

### 6. 相同节点在不同管理面上的规则不一致

影响：

- `ProjectTreeWidget`、项目树管理对话框和 DataPage 节点管理对“普通子文件夹是否受保护”的理解不同。
- 后续若只修一个入口，另一个入口会继续保留旧行为，回归风险很高。

## 推荐优化方向

### A. 收口可视化页的渲染抽样接口

建议原则：

- `ChartPage` 内所有 `decimate_xy_for_rendering()` 调用统一改为 policy 风格，不再混用旧关键字参数。
- 若需要兼容旧接口，也应在 `core/rendering.py` 提供显式过渡层，而不是让页面层继续自行拼接参数。

建议补充守护：

- 大点数预览路径窄测
- 常规 `_redraw_now()` 路径窄测
- 抽样返回值数量与消费方式一致性检查

### B. 建立“根分组文件夹”和“普通子文件夹”的共享判定 helper

建议统一明确：

- 何时一个 folder 是 root group
- 何时一个 folder 只是隶属于某集合的普通子文件夹
- 哪些能力由“是否根分组”决定
- 哪些能力由“所属集合”决定

至少需要共享的判断结果：

- `is_root_group_folder`
- `folder_collection_group`
- `is_protected_folder`
- `display_folder_name`
- `folder_move_targets`

### C. 扩展 folder 节点的完整动作矩阵

普通子文件夹应支持：

- 重命名
- 删除
- 移动到同集合其它 folder
- 批量删除
- 与同类 folder 的批量移动 / 批量拖动移动

根分组文件夹仍应保持：

- 不可重命名
- 不可删除
- 不可跨集合移动
- 保留集合级导入、新建子文件夹、清理空子文件夹等入口

### D. 重做项目树批量操作的公共 payload 契约

建议把批量动作从“只支持 `series` / `curve`”升级为“支持同类、同项目、兼容目标一致的节点集合”。

建议统一前置校验：

- 节点 kind 必须一致
- 项目必须一致
- 目标集合必须兼容
- 目标候选应取交集而不是并集
- 命名冲突、父子循环、非法目标应在执行前整体拒绝，而不是移动到一半才报失败

### E. 拆分 `source_file` 的拖放语义

建议明确三类路径：

- 拖到 `data_file` / 数据集 folder：导入语义
- 拖到数字化 folder：导入语义
- 拖到源文件 folder：节点移动语义

同时需要保证：

- 右键“移动到...”与拖动移动使用同一目标解析逻辑
- 源文件集合内部移动不会触发导入对话框

### F. 把项目级命令从 MainWindow 壳层抽成可复用入口

当前 `MainWindow` 已有：

- `_save_current_project_from_panel()`
- `_close_current_project_from_panel()`

但项目根右键菜单新增“保存项目 / 另存项目 / 关闭项目”后，至少还需要：

- 显式“另存项目”命令
- 可被项目树、项目树管理对话框复用的项目级 action/service
- 与当前项目上下文严格绑定的调用约束

### G. 为 root-scoped 清理和多入口一致性补守护

建议优先补充：

- `remove_empty_folders(root_id=None)` 的 UI 入口测试
- 项目根节点上下文菜单触发的清理测试
- 项目树管理对话框中的同路径测试
- 普通子文件夹在项目树、项目树管理、DataPage 节点管理中的名称与动作一致性测试

## 验收标准

- 从项目树激活数据文件/数据列进入可视化页时，绘图不再触发 `decimate_xy_for_rendering(..., max_points=...)` 报错。
- 普通子文件夹在项目树右键菜单和项目树管理中均可重命名、删除、移动。
- DataPage 节点管理对普通子文件夹显示真实节点名；根分组仍保持集合语义展示。
- 同类节点的批量删除、批量移动、批量拖动移动在允许范围内可用，并在非法目标下整体拒绝。
- 源文件节点在源文件集合内支持拖动移动，拖到数据集/数字化集合时仍按导入处理。
- “清理空文件夹”根级入口恢复可用，并有 `root_id=None` 路径守护。
- 项目根节点菜单包含“保存项目 / 另存项目 / 关闭项目”，并在项目树管理中同步可用。

## 本阶段不纳入的范围

- 项目树整体视觉重设计
- 跨项目批量移动能力
- 全量 UI 自动化回放测试
- 可视化页大规模性能重构
- DataPage / MainWindow 的大规模结构重拆

## 风险与回退

风险：

- 若继续把 `group_type` 直接等同于“受保护根分组”，普通子文件夹问题会在更多入口重复出现。
- 若批量移动仍沿用“部分执行后再报错”的模式，用户数据结构会进入半成功状态。
- 若项目级命令只接到某一个菜单入口，项目树管理与主树很快又会再次漂移。

回退方式：

- 先抽共享 helper 和批量前置校验，再接菜单和拖放入口。
- 先修渲染接口漂移与核心能力矩阵，再补多入口 UI 行为。
- 对“清理空文件夹”优先验证 UI 入口与上下文绑定，不重复重写底层删除 service。
