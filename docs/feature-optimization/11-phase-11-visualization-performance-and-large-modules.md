# Phase 11：可视化性能、主题切换与超大模块后续治理

## 目标与完成定义

目标：

- 收口当前仍然最影响体验的性能热点，尤其是图表页的大曲线重绘与主题切换卡顿。
- 控制仍然偏大的页面文件继续膨胀，避免后续优化阶段再把结构问题带回来。

完成定义：

- 图表页对“大曲线重绘”和“主题切换”的路径区分更明确。
- 超大页面的后续拆分边界被正式定义，不再靠临时抽 support/helper 维持。
- 与性能相关的窄测和观测点具备最小闭环。

## 当前审查结论

当前仓库里仍然存在明确的热点：

- `ui/pages/chart_page.py`
  - 约 4300+ 行
  - 已有 decimated 预览，但完整 `_redraw_now()` 仍然承担大量同步工作
  - 主题切换、大曲线样式变化、扩展 patch 应用仍容易合并到同一条重路径
- `ui/pages/data_page.py`
  - 约 4900+ 行
  - 导入、预览、扩展、资产应用和树联动仍集中在同一页
- `ui/pages/digitize_page.py`
  - 约 3600+ 行
- `ui/pages/analysis_page.py`
  - 约 2900+ 行
- `ui/pages/process_page.py`
  - 约 1900+ 行
- `ui/pages/settings_page.py`
  - 约 1600+ 行

这说明前一轮重构已经做了大量拆分，但“剩余核心工作台页面”仍然是后续维护与性能治理的主战场。

## 主要问题

### 1. 图表页主题切换与大曲线重绘仍偏同步

影响：

- 主题切换时容易走完整绘图刷新链路。
- 大曲线存在 decimated 预览，但完整渲染与样式合并仍可能造成卡顿。

### 2. 图表页职责仍偏重

影响：

- 样式状态、扩展 patch、列表交互、绘图渲染、导出与主题响应耦合度仍高。
- 后续再叠加功能，极易继续引入回归。

### 3. DataPage 等超大页面仍有继续膨胀风险

影响：

- 导入预览、扩展配置、资产入口、树联动和批处理逻辑仍然集中。
- 单页文件过大时，局部优化很容易演化成接口漂移或重复状态。

### 4. 缺少性能基线与轻量观测点

影响：

- 目前“是否卡顿”更多依赖人工体感。
- 难以区分是：
  - 数据量问题
  - 扩展 patch 合并问题
  - 主题切换引发的重复重绘问题

## 推荐优化方向

### A. 区分主题刷新与数据重绘路径

建议优先把图表页重绘路径拆成两类：

- 数据内容变化
  - 曲线集、扩展输出、轴配置、样式配置变化
- 主题/展示变化
  - 亮暗主题切换
  - 面板配色或纯展示层变化

目标是避免主题切换无差别触发完整数据重算。

### B. 为图表页补完真正的阶段化渲染

当前已有 decimated 预览基础，建议进一步明确：

- 快速预览阶段
- 完整渲染阶段
- 过期结果保护
- 可取消或可丢弃的旧渲染任务

这部分应与已有 `TaskManager` / `BackgroundTask` 方向对齐，而不是继续堆同步分支。

### C. 明确剩余大页面的拆分边界

建议优先定义而非立刻全面重写：

- `DataPage`
  - 导入工作流
  - 预览工作流
  - 扩展/配置资产工作流
  - 树联动与结果落盘工作流
- `ChartPage`
  - 渲染管线
  - 曲线列表/上下文菜单
  - 样式与扩展状态面板
- `DigitizePage`
  - 交互编辑
  - 导出/保存
  - 辅助工具状态

目标是先冻结边界，再进入下一轮收敛式拆分。

### D. 增加性能窄测与观测点

建议优先建立：

- 大曲线点数阈值下的渲染耗时日志
- 主题切换前后重绘次数/重路径统计
- 图表页大输入下的旧任务丢弃验证

不要求做完整 benchmark 平台，但至少要能在本地复核“有没有明显退化”。

## 验收标准

- 图表页主题切换不再无差别触发完整重绘链路。
- 大曲线渲染路径具有更明确的预览/完整阶段边界。
- `DataPage`、`ChartPage`、`DigitizePage` 的下一轮拆分边界在文档和代码组织上保持一致。
- 后续优化前，可以通过轻量观测点判断是否出现性能退化。

## 本阶段不纳入的范围

- 全部页面再次大规模重构
- 全量 benchmark 基础设施
- 渲染引擎彻底替换
- 跨进程绘图执行

## 风险与回退

风险：

- 若直接再次大规模拆页面，容易与当前功能优化阶段目标冲突。
- 若只做性能小修不冻结边界，后续代码仍会继续回流到 monolith 页面中。

回退方式：

- 先把主题刷新路径、渲染阶段和拆分边界定义清楚。
- 先做最影响体验的图表页，再逐步处理 DataPage / DigitizePage 等剩余热点。
