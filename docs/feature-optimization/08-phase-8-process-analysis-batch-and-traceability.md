# Phase 8：处理与分析工作台的批量执行与结果可追溯性

## 目标与完成定义

目标：

- 提升 process page 和 analysis page 的长链路工作流效率。
- 让用户在多输入、多轮调参、保存结果和导出报告时，更容易理解“结果从哪里来、还能否复跑、是否便于比较”。

完成定义：

- 批量执行、复跑、结果对比和保存后的来源摘要明显改善。
- 用户不需要再靠记忆推断当前结果对应的输入、参数和模板。

## 当前审查结论

process page 当前已具备：

- 多输入
- pipeline 编辑
- pipeline 模板
- 保存目标选择
- 批量导出

analysis page 当前已具备：

- 多输入
- 分析扩展参数
- 结果 tab
- 报告模板
- 结果保存与导出

这说明基础功能已齐，但工作流层面还偏“单步功能集合”，不是“可批量、可比较、可追溯”的成熟体验。

## 主要问题

### 1. 复跑与结果对比能力不足

影响：

- 用户多次调参后，不容易快速比较几轮结果差异。
- 对 pipeline / analysis 扩展来说，复跑体验还不够显式。

### 2. 批量工作流仍偏局部

影响：

- process page 虽已支持部分批量导出，但没有形成统一的批量执行和保存摘要。
- analysis page 对多输入结果的保存、命名和比较仍有提升空间。

### 3. 结果可追溯性不足

影响：

- 用户保存结果后，需要额外回忆：
  - 用了哪些输入
  - 哪套参数
  - 哪个模板
  - 是否来自某个 pipeline preset

## 推荐优化方向

### A. 统一“复跑”语义

建议显式提供：

- 复跑当前配置
- 使用上一次输入重新运行
- 复制当前配置为新试验

### B. 批量执行摘要

建议对批量处理/分析提供统一摘要：

- 输入数量
- 结果数量
- 跳过项
- 告警项
- 保存位置

### C. 结果对比能力

建议优先做轻量能力：

- 结果 tab 间快速切换与比较
- 当前结果与上一次结果的摘要对比
- 图表/表格层面的最小对比入口

### D. 保存结果时的来源摘要

建议在保存或导出时记录并显示：

- 输入摘要
- 参数摘要
- 模板摘要
- 执行时间

重点是先让结果“解释得清楚”，而不是先做复杂实验管理系统。

## 验收标准

- process / analysis 页对多轮调参和多输入执行更友好。
- 保存或导出后的结果更容易追溯到输入、参数和模板。
- 用户能更方便地复跑和比较结果。

## 本阶段不纳入的范围

- 完整实验管理平台
- 项目外部协作审阅系统
- 云端任务队列

## 风险与回退

风险：

- 若把 provenance 设计为重型结构，会扩大持久化与 UI 改动面。
- 若批量执行和结果对比一次性做得太重，容易与 Phase 5 的后台执行耦合过深。

回退方式：

- 先做轻量来源摘要、复跑入口和批量结果摘要。
- 先在 process/analysis 内部分别做好，再逐步统一共享模型。
