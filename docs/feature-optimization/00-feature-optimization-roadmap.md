# 功能优化路线图

## 目标

在不重新打开大规模结构重构的前提下，优先改善用户已经能感知到的功能体验问题，尤其是：

- 主题切换后的首页视觉一致性
- 可视化页的可见性批量操作效率
- 绘图扩展与样式设置之间的可预期性
- 扩展协议、注册、兼容性与设置面的长期可维护性
- 大曲线和长耗时操作的交互反馈、后台执行与可取消性
- 数据导入、来源追踪、模板/预设复用以及处理/分析批量工作流
- AI 模块的禁用收口与后续重做准备

## 功能问题清单

### 1. 主页背景图未真正按主题切换

现状：

- `_HomeBannerWidget` 当前固定加载 `assets/aline_home_background.png`。
- `paintEvent()` 中确实按 `isDarkTheme()` 切换了遮罩渐变，但暗色主题没有切换到独立暗色背景资源。
- 仓库中已经存在 `assets/aline_home_background_dark.png`，但未接入。

影响：

- 暗色主题下首页背景层次与亮色主题不对称。
- 当前方案依赖强遮罩修正，容易让亮暗主题在视觉语义上不一致。

### 2. 可视化页曲线列表的批量可见性工作流不完整

现状：

- 曲线列表支持多选。
- 工具栏按钮 `_toggle_selected_visibility()` 已支持按选中批量切换可见性。
- 右键菜单仅提供：
  - 重命名显示名称
  - 恢复原始名称
  - 仅显示选中

影响：

- 用户能通过工具栏做批量隐藏/显示，但右键菜单没有显式语义动作，发现成本高。
- 当前“仅显示选中”属于单向动作，缺少“显示已选中”“隐藏已选中”“全部显示”等补全动作。
- 多选时上下文菜单的操作语义不完整。

### 3. 绘图扩展会覆盖曲线样式/绘图样式设置

现状：

- 图表页当前通过 `PlotExtensionContext` 记录：
  - `figure_state_patch`
  - `plot_style_patch`
  - `curve_style_patches`
- `ChartPage._redraw_now()` 在 `before_plot` 阶段先收集扩展 patch，再通过 sequence 与手动版本号合并得到有效样式。
- 这个机制在实现层面是合理的，但用户侧没有清晰的“谁应该赢”的规则说明，也没有显式控制开关。

影响：

- 用户会把“我在曲线样式/绘图样式面板里改了值，却又被扩展改掉”理解为 bug。
- 当前冲突不容易定位，因为用户看不到：
  - 哪个扩展改了哪些字段
  - 当前最终生效值来自手动设置、模板还是扩展

### 4. 扩展目标曲线与影响范围的可视反馈不足

现状：

- 当前绘图扩展支持绑定目标曲线和参数。
- 但列表、样式面板、扩展状态区缺少“该扩展正在影响哪些字段/哪条曲线”的明确展示。

影响：

- 用户容易把样式冲突误判为普通 UI 刷新问题。
- 批量图表场景下，扩展目标不透明会放大误操作成本。

### 5. 曲线可见性缺少恢复类动作

现状：

- 有“仅显示选中”和批量切换可见性。
- 但没有显式“全部显示”“恢复上一次隐藏前状态”“仅隐藏选中以外”这类恢复型动作。

影响：

- 多轮筛选后，用户容易丢失全局可见性状态。
- “仅显示选中”之后恢复完整视图不够直接。

### 6. 扩展协议对内部场景够用，但对后续扩展生态仍偏“实现驱动”

现状：

- 当前扩展采用 Python 文件扫描 + `register_extensions(registry)` 入口。
- 四类扩展 dataclass、registry 注册方法、load report 和 contract validation 已经形成基础闭环。
- 内置接口示例扩展和 `extensions/README.md` 也说明当前协议已可用。

问题：

- 协议元数据仍偏分散，plot/process/analysis/digitize 的 capability 不能统一声明。
- 没有显式 API 兼容范围、authority、warnings/progress/cancel 契约。
- UI 看到的是 `build_extension_entry()` 生成后的结果，但扩展作者侧的“声明模型”和 UI/runtime 侧的“消费模型”还不是同一个抽象。
- 长期看，`register_extensions(registry)` 这种纯命令式注册方式不利于做预检查、兼容提示、批量迁移和生态扩展。

### 7. 大曲线/长耗时工作流仍缺少后台执行与进度反馈

现状：

- chart page 已有 decimation。
- process page 主要依赖 `QTimer` 防抖，仍然同步跑 pipeline。
- analysis page 仍然同步跑分析。

问题：

- 大输入时 UI 仍可能阻塞。
- 用户看不到当前正在计算什么、还要多久、能否取消。
- 后续若扩展能力继续增强，这个缺口会更明显。

### 8. 数据导入、预览与来源追踪工作流分散

现状：

- 项目树右键菜单支持导入源文件、导入数据文件、导入数字化图片。
- DataPage 也有自己的导入、预览和 pending import 逻辑。
- source file、dataset、series、analysis result 之间存在实际来源关系，但用户面看到的链路仍偏离散。

问题：

- 同类导入动作在不同入口上的行为口径仍有继续统一的空间。
- 大批量导入时缺少更强的预检、命名策略和导入摘要。
- 后续追踪“某数据列来自哪个源文件/导入参数/处理链”还不够直观。

### 9. 全局模板/预设/扩展配置的资产工作流偏基础

现状：

- `global_assets` 已管理 pipelines、figure templates、report templates、curve style templates、plot themes、extension configs、AI 资源。
- settings/process/analysis/chart 多处可以使用这些全局资产。

问题：

- 当前更偏“能存、能列、能改”，还不是完整的资产库体验。
- 缺少更统一的预览、复制、导入/导出、默认值切换、依赖提示和使用范围说明。

### 10. 处理与分析工作台的批量工作流和结果可追溯性仍有提升空间

现状：

- process page 已支持多输入、pipeline 模板、保存目标选择、批量导出。
- analysis page 已支持多输入、报告模板、结果保存和导出。

问题：

- 缺少更明确的复跑、结果对比、批量执行、输入快照和来源摘要。
- 结果保存后，用户仍需要额外推断这份结果来自哪组输入、哪套参数、哪版模板。

### 11. AI 模块当前更适合先禁用收口，再单独重做

现状：

- `ai.command_registry` 已成为 AI 命令定义唯一源。
- `ai.command_layer` 已收口为 dispatcher，但仍承接旧兼容导出。
- `ai.agent`、settings 中的 AI 工具管理以及历史导入点之间仍存在兼容漂移风险。
- 主窗口历史上保留过 AI 启动期导入，即使当前用户面并未稳定启用 AI，也会把应用启动绑到 AI 模块状态上。

问题：

- AI 模块目前对普通用户主流程不是刚需。
- 继续在现有结构上做局部修补，容易让“命令面 / agent / 配置 / 工具管理 / UI入口”继续半耦合、半停用。
- 若后续准备彻底重做 AI 模块，现在更适合先把旧入口安全下线。

## 阶段划分

### Phase 1：主页与曲线列表交互抛光

处理低风险、高收益、用户立刻可感知的功能点：

- 首页按主题切换背景图
- 曲线列表右键菜单补全隐藏/显示相关动作
- 统一工具栏与右键菜单的可见性命令口径

### Phase 2：绘图样式优先级与扩展契约

处理可视化页最核心的行为一致性问题：

- 定义手动样式、模板样式、扩展 patch 的优先级
- 区分“扩展建议值”和“扩展强制值”
- 明确哪些扩展允许覆盖用户手动设置

### Phase 3：可视化工作流与反馈增强

在 Phase 2 契约稳定后补全用户反馈：

- 扩展影响范围可视化
- 曲线显示状态恢复动作
- 样式冲突提示与说明

### Phase 4：扩展协议与注册机制现代化

在不推翻当前四类扩展体系的前提下，补齐长期生态需要的协议元数据和注册抽象：

- 兼容性版本与 capability 声明
- style authority / long-running / progress / cancel 等元数据
- 更统一的扩展声明模型
- 更适合预检查和设置页消费的注册信息结构

### Phase 5：大曲线与长耗时任务的异步化和渐进反馈

把性能优化从“算法和 decimation”推进到“用户工作流体验”：

- chart/process/analysis 页的后台执行
- 进度提示、取消、阶段性结果
- 大曲线与大批量输入下的渐进式预览和结果回填

### Phase 6：数据导入、预览与来源链路优化

围绕 DataPage、项目树导入入口和 source file -> dataset -> downstream workflow 做统一：

- 导入预检与批量导入摘要
- 预览参数与导入参数的一致性
- 命名策略、目标位置建议和来源追踪

### Phase 7：全局模板/预设/扩展配置资产库优化

把 `global_assets` 从“存储层”进一步提升到“可复用资产工作流”：

- 主题/模板/样式/扩展配置的预览和应用路径
- 复制、导入/导出、默认值切换、依赖提示
- 设置页与业务页之间的统一资产使用体验

### Phase 8：处理与分析工作台的批量执行与结果可追溯性

在前几阶段稳定后，统一 process/analysis 两条长链路的执行体验：

- 复跑、批量运行、结果对比
- 输入快照、参数摘要、模板来源记录
- 保存/导出后的结果可追溯性

### Phase 9：AI 模块禁用收口与重做准备

在不影响主业务页的前提下，把旧 AI 模块从启动链路和普通用户路径中解耦：

- 清理主窗口和常规页面中的启动期 AI 硬依赖
- 明确哪些旧 AI 入口先禁用、隐藏或只保留兼容壳
- 保留最小兼容层，避免测试/历史调用路径立刻全面失效
- 为后续全新 AI 模块设计预留独立 phase 入口

## 执行原则

- 优先补齐用户能直接感知的缺口，再做复杂规则。
- 功能优化不得破坏已经通过的重构护栏与窄测。
- 对可视化页的任何优化，都应避免再次把样式、扩展、列表菜单逻辑进一步耦合回单一函数。
- 对扩展协议的优化应优先“增量补契约”，而不是直接推翻现有四类扩展实现。
- 对大曲线和长耗时任务的优化，应同时关注“算得快”和“用户知道正在发生什么”。
- 对 AI 模块，应优先“先脱离主流程、再整体重做”，而不是继续把半稳定旧入口绑在主窗口启动链路上。
