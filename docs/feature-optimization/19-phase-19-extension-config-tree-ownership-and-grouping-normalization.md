# Phase 19：扩展配置树归属与分组收口

## 目标与完成定义

目标：

- 修复项目树中扩展配置的组织方式，使配置项按具体扩展节点归属管理，而不是全部直接扁平挂在扩展类别节点下。
- 收口扩展配置在项目树、管理入口与后续操作中的“所有权语义”，避免用户只能先理解类别、再手动辨认配置属于哪个扩展。

完成定义：

- 扩展配置树节点应先按扩展节点分组，再展示该扩展名下的配置项。
- 用户可以从树结构上直接看出某个配置属于哪个扩展，而不必在同类配置列表中人工辨认。
- 与扩展配置相关的创建、复制、设为默认、导出、删除等动作，均应围绕“具体扩展”这一归属维度展开。
- 项目树管理与右键菜单的显示结构、命名和行为保持一致。
- 补齐扩展配置树归属收口的窄范围测试，不做大规模回归。

## 当前审查结论

该问题属于树模型语义与扩展配置所有权模型没有完全对齐，而不是单纯的展示顺序问题：

- 当前扩展配置仍主要按扩展类别聚合，例如 processing / analysis / plot / digitize。
- 在每个类别节点下，多个扩展的配置被直接并列展示，具体配置与其所属扩展之间缺少中间归属层。
- 当同一类别下存在多个扩展、每个扩展又有多套配置时，用户需要依赖命名猜测配置归属，树结构本身无法提供充分上下文。
- 这种扁平挂载方式也会放大后续命令设计的复杂度，因为“新建配置”“复制配置”“设为默认”等动作其实都更接近“针对某个扩展”的管理，而不是针对整个类别的无差别管理。

用户当前反馈已经明确说明：

- 扩展配置应按扩展节点管理。
- 现有“全部直接挂在扩展类别节点下”的组织方式不符合管理预期，应整体修复，而不是只调整文案。

## 主要问题

### 1. 扩展配置缺少明确的扩展归属层

影响：

- 用户在项目树中无法直接建立“扩展 -> 配置”的映射关系。
- 同类扩展较多时，配置节点会迅速失去可读性。

### 2. 树结构语义与管理动作语义不一致

影响：

- 树上按类别浏览，动作上却往往需要先确定具体扩展，交互链路不自然。
- 后续新增“按扩展创建配置”“查看默认配置”“迁移配置”等动作时，会不断遇到归属上下文缺失的问题。

### 3. 项目树与配置管理入口容易继续漂移

影响：

- 若只改树展示、不同步命令入口和管理对话框，最终会出现树上一套结构、弹窗里另一套结构。
- 扩展配置的默认项、内置项和用户项之间的关系也更容易被错误表达。

### 4. 当前测试难以守住扩展配置树归属契约

影响：

- 即使短期内通过手工调整显示顺序解决视觉问题，也无法防止后续改动再次退化成按类别扁平挂载。
- 没有围绕“扩展节点分组”建立测试，后续重构时很难快速识别回归。

## 推荐优化方向

### A. 明确扩展配置树的归属模型

建议统一定义：

- 第一层：扩展类别
- 第二层：具体扩展节点
- 第三层：该扩展名下的配置项

建议原则：

- 类别仍可保留，负责表达能力域。
- 具体扩展节点负责表达配置所有权。
- 配置项不再直接越过扩展节点扁平挂载。

### B. 让管理动作围绕具体扩展展开

建议至少同步以下动作语义：

- 新建配置
- 复制配置
- 导出配置
- 设为默认
- 删除配置

建议要求：

- 这些动作应能够明确知道当前操作的是哪个扩展。
- 不再让类别节点承担过多需要具体扩展上下文的管理职责。

### C. 统一项目树与管理对话框的结构

建议同步检查：

- 项目树右键菜单
- 扩展配置相关管理对话框
- 可能存在的扩展配置选择器或预览入口

要求：

- 展示层级一致
- 命名口径一致
- 默认配置与普通配置的区分方式一致

### D. 明确默认配置与扩展归属的关系

至少需要明确：

- 默认配置是否始终从属于具体扩展
- 默认配置在树上的展示是否仍置顶
- 当某个扩展没有用户配置时，是否仍保留扩展节点

推荐最低要求：

- 默认配置仍按扩展归属展示
- 具体排序策略可以延续现有默认项优先，但不能破坏“扩展先于配置”的层级关系

### E. 补齐窄范围测试守护

建议优先补充：

- 扩展配置树节点按扩展分组的结构测试
- 同类别多扩展场景下的显示测试
- 创建/复制/设为默认等动作是否保留扩展上下文的测试
- 项目树与管理入口结构一致性的测试

## 验收标准

- 项目树中的扩展配置不再直接扁平挂在类别节点下，而是先进入对应扩展节点再展示配置项。
- 用户可以从树结构上直接识别配置所属扩展。
- 扩展配置相关动作在项目树和管理入口中都围绕具体扩展上下文执行。
- 默认配置与普通配置的展示顺序可读且不破坏扩展归属层级。
- 相关修改具备窄范围自动化测试，不依赖大规模回归即可守住该结构契约。

## 本阶段不纳入的范围

- 扩展协议版本体系重做
- 全局资产页整体信息架构改版
- 扩展配置文件格式大规模迁移
- AI 扩展配置体系重做

## 风险与回退

风险：

- 若只在 UI 层临时插入一层展示节点，而不把管理动作的扩展上下文一起收口，后续依旧会出现“显示像分组了，操作仍是扁平”的漂移。
- 若默认配置、内置配置和用户配置的排序规则不清晰，树层级调整后容易引发名称冲突或默认项识别混乱。
- 若项目树和管理对话框不同步修改，用户会在两个入口看到不同的结构模型。

回退方式：

- 优先将回退范围限制在扩展配置树构建与配置管理入口，不回退扩展注册和配置持久化本身。
- 若阶段实施中发现完整分层超出当前修改面，可先保证“树结构与动作上下文统一按扩展归属”这一最低闭环，再决定是否延后更细的排序或预览优化。
