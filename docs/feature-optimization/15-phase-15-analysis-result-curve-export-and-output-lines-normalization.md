# Phase 15：分析结果曲线导出与输出线契约规范化

## 目标与完成定义

目标：

- 修复分析页 `curve_fit` 等分析类型在“导出结果曲线”入口上的异常行为，恢复结果曲线导出能力。
- 收口分析结果 `lines`、`plot_series`、按钮可见性与当前 tab 结果之间的契约，避免“结果存在但导不出”或“按钮状态与结果内容不一致”的情况继续出现。

完成定义：

- `curve_fit` 结果在分析完成后，`导出结果曲线` 按钮可见且可用。
- `curve_fit`、`peak_detect`、`error_compare` 等具备输出线的分析类型，统一通过稳定的 `lines` / `plot_series` 契约驱动导出。
- `_analysis_result_lines()`、结果 tab 恢复逻辑、导出逻辑与按钮状态计算保持一致，不再依赖临时补丁或分散兼容。
- 增加针对分析结果曲线导出的窄测试，覆盖当前结果、保存后恢复结果和多 tab 切换场景。

## 当前审查结论

phase12-14 收尾后，项目树分组与导出目标归属问题已基本收口，但相邻复查确认分析页仍有一类独立异常：

- `tests/test_ui.py` 中现有 `curve_fit` 结果曲线导出用例失败，按钮可见性和实际导出结果并不稳定。
- `AnalysisPage` 当前同时存在：
  - `_analysis_result_lines()`
  - `result["lines"]`
  - `result["plot_series"]` / `result["_plot_series"]`
  - 结果 tab 恢复时的 `setdefault("lines", ...)`
- 这些来源并没有形成单一输出线契约，导致：
  - 某些分析结果已经有图表展示，但按钮仍隐藏
  - 某些恢复后的结果 tab 才补出 `lines`
  - 导出逻辑对当前结果、恢复结果和不同分析类型的行为不完全一致

这类问题已经超出 phase14 的“项目树分组语义”范围，更适合作为下一阶段单独治理。

## 主要问题

### 1. 分析结果输出线来源不唯一

影响：

- 按钮显隐、结果预览和导出行为可能基于不同字段判断。
- 新增分析类型时，容易只补图表展示，不补导出契约。

### 2. `curve_fit` 恢复路径与即时运行路径可能不一致

影响：

- 即时运行后和从已保存结果恢复后，`lines` / `_plot_series` 的内容可能不同。
- 导出结果曲线在 tab 切换或恢复后容易失效。

### 3. 缺少针对“分析结果导出曲线”的专门守护

影响：

- 目前窄测覆盖了导出目标分组，但还没有充分冻结输出线契约本身。
- 后续改动分析结果摘要、图表或恢复逻辑时，容易再次破坏导出路径。

## 推荐优化方向

### A. 建立分析结果输出线的单一契约

建议统一明确：

- 哪些分析类型必须产出 `lines`
- `plot_series` 与 `lines` 的对应关系
- `_analysis_result_lines()` 是否作为唯一读取入口
- 按钮可见性、导出和图表渲染是否统一依赖该入口

### B. 收口即时结果与已保存结果的恢复逻辑

建议保证：

- 即时运行生成的结果与恢复后的结果拥有一致的输出线结构
- `curve_fit`、`error_compare` 等内建分析在恢复时不再额外补丁式 `setdefault`
- 当前 tab 与后台保存结果使用同一套标准化 helper

### C. 为导出结果曲线增加窄测试

建议优先覆盖：

- `curve_fit` 结果导出单曲线
- `peak_detect` 多输出线选择后导出
- 恢复已保存结果后导出
- 多 tab 切换后导出当前 tab 结果
- 无输出线分析类型保持按钮隐藏

## 验收标准

- `curve_fit` 结果曲线导出恢复可用。
- 按钮状态与分析结果输出线契约一致。
- 即时结果与恢复结果的输出线结构一致。
- 新增窄测覆盖分析结果曲线导出核心路径。

## 本阶段不纳入的范围

- 分析页整体视觉重做
- 报告模板系统重构
- 大规模扩展分析类型
- 全量分析页自动化回放测试

## 风险与回退

风险：

- 若继续依赖多个结果字段并行存在，未来每次补新分析类型都可能再次打破导出契约。
- 若只修按钮显隐而不收口结果结构，导出链路仍会在恢复或多 tab 场景中反复回归。

回退方式：

- 先明确单一输出线读取入口。
- 再按分析类型逐步收口即时结果与恢复结果的结构差异。
