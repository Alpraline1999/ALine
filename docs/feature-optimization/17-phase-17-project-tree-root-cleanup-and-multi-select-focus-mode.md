# Phase 17：项目树根级清理与多选专注模式

## 目标与完成定义

目标：

- 修复项目树 root-scoped 的“清理空文件夹”功能，使其与“清理空子文件夹”形成清晰且可验证的双层契约。
- 让项目树专注模式从“仅支持单节点”升级为“支持多选节点联合专注”，并同步覆盖项目树管理对话框与上下文菜单。

完成定义：

- 从项目树树作用域入口触发“清理空文件夹”后，可正确清理当前项目中的空普通文件夹；受保护根分组文件夹仍不会被误删。
- “清理空子文件夹”继续只作用于指定文件夹子树，两类清理动作的作用域、反馈文案和刷新行为保持一致。
- 项目树支持对多个选中节点进入专注模式，专注后保留所选节点、它们的祖先链及其子树，并正确隐藏其余分支。
- 项目树管理对话框、右键菜单和项目树主体对专注状态的按钮文案、启用条件和退出逻辑保持一致。
- 补齐 root-scoped 清理与多选专注模式的窄范围测试，不依赖全量 UI 回归。

## 当前审查结论

这两个问题都属于项目树交互契约未收口，而不是单个 API 的偶发失效：

- `app/project_tree_command_service.py`
  - `prune_empty_folders(root_id=None)` 已直接委托 `project_manager.remove_empty_folders(root_id)`，接口表面支持根级清理。
- `core/project_tree_service.py` / `core/project_manager.py`
  - 后端已有 `remove_empty_folders(root_id=None)` 能力，既有测试主要覆盖“指定 root_id 的子树清理”，未覆盖树作用域 UI 入口。
- `ui/widgets/project_tree.py`
  - 树作用域菜单的“清理空文件夹”直接绑定 `_cmd_prune_empty_folders`，但当前 UI 测试仅覆盖菜单展示与带 `root_id` 的反馈路径。
  - 专注状态只由单个 `_focused_item_key` 表示，`can_focus_selected_item()` 也强制要求仅选中 1 项，因此多选节点天然无法进入专注。
  - `_focus_visible_keys()` 当前只按单个焦点节点构造可见集合，无法表达多个节点联合可见。
- `ui/dialogs/project_tree_manage_dialog.py`
  - 专注按钮逻辑直接依赖单个 `_focused_item_key`，按钮文案和启用状态均围绕“单节点匹配”设计。
- `tests/test_ui.py`
  - 现有专注模式测试仅覆盖单节点专注与退出专注。
  - “清理空文件夹”只验证菜单中存在该动作，功能测试只覆盖传入具体 `root_id` 的子树清理反馈。

用户反馈说明：

- “清理空子文件夹”正常，说明子树路径基本可用，问题集中在 root-scoped 入口契约、项目上下文、刷新选择或 UI 测试缺口。
- “应支持多选节点专注模式”说明当前单节点实现已成为交互能力上限，且项目树管理对话框也会同步受限。

## 主要问题

### 1. 根级清理与子树清理没有形成明确的双层行为契约

影响：

- 用户可以看到“清理空文件夹”入口，但当前行为不可靠，容易被误判为功能无效。
- 后端已具备能力时，UI 层失效会让后续维护更难定位责任边界。

### 2. 专注模式状态模型被硬编码为单节点

影响：

- 多选场景下无法进入联合专注，和项目树已支持的多选删除、多选移动能力不对称。
- 一旦在项目树管理对话框中多选，专注按钮只能退化为禁用或错误文案。

### 3. 项目树主体、管理对话框与右键菜单对专注状态的理解不一致

影响：

- 即使只在 `ProjectTreeWidget` 内部补一个多选可见集合，管理对话框仍会沿用旧的按钮判定，继续表现异常。
- 若只更新按钮文案而不调整树内部状态模型，又会引入新的“看起来可用、实际无效”的回归。

### 4. 当前测试覆盖无法守住这两条用户主链路

影响：

- root-scoped 清理缺少直接验证，导致 phase16 虽然补了入口，但无法保证真正可用。
- 多选专注完全没有测试基线，后续改动容易只做 UI 文案不做真实行为闭环。

## 推荐优化方向

### A. 明确“清理空文件夹”与“清理空子文件夹”的作用域契约

建议统一定义：

- “清理空文件夹”：
  - 作用于当前项目整棵树中的空普通文件夹。
  - 不删除受保护根分组文件夹。
  - 由树作用域入口、项目根节点菜单和项目树管理对话框共享同一命令路径。
- “清理空子文件夹”：
  - 只作用于某个文件夹节点之下的子树。
  - 继续保留在文件夹右键菜单中。

建议补充守护：

- `root_id=None` 的 UI 功能测试
- 带选中节点/带专注状态时的刷新与选择恢复测试
- 无可清理项时的成功反馈一致性测试

### B. 将专注状态从单个 key 升级为“专注节点集合”

建议原则：

- 状态层不再只保存单个 `_focused_item_key`，而应能表达多个节点的联合专注集合。
- 可见集合应取每个专注节点的：
  - 祖先链
  - 节点自身
  - 子树
- 多选专注后的隐藏规则应以“联合可见集合”计算，而不是退化为第一个节点。

### C. 统一多选专注的交互入口

建议至少同步以下入口：

- 项目树右键菜单：
  - 单选时保留“专注此处 / 切换专注到此处”
  - 多选时新增或切换为“专注所选”
- 项目树管理对话框：
  - 多选时允许进入专注
  - 若当前选择与专注集合一致，则按钮显示“退出专注”
- 公共 API：
  - 需要有能够表达“专注当前选中集合”的统一调用，而不是让各入口自行拼装状态

### D. 明确多选专注的边界策略

至少需要明确：

- 是否允许跨多个项目根同时专注
- 是否允许包含全局资源节点
- 当专注集合中的某些节点在删除/移动/清理后消失时，如何自动降级或退出专注

推荐最低要求：

- 至少支持同一项目内的多选专注
- 若选择跨项目或跨全局资源，必须给出明确规则，不能静默退化为单节点

### E. 补齐窄范围测试守护

建议优先补充：

- root-scoped “清理空文件夹” UI 行为测试
- root-scoped 清理后的树刷新与反馈测试
- 多选专注的联合可见性测试
- 多选专注退出后树状态恢复测试
- 项目树管理对话框按钮文案/启用条件测试

## 验收标准

- 项目树树作用域入口触发“清理空文件夹”后，当前项目中的空普通文件夹会被正确移除。
- “清理空子文件夹”继续正常工作，且不会和根级清理互相混淆。
- 选中多个节点后可进入专注模式，专注视图保留所选节点相关分支并隐藏无关分支。
- 项目树管理对话框与右键菜单对多选专注的按钮文案、启用条件和退出逻辑一致。
- 相关变更具备窄范围自动化测试，不需要大规模回归测试即可守住该能力。

## 本阶段不纳入的范围

- 项目树整体视觉改版
- 专注模式与筛选模式的大规模统一重构
- 跨页面共享导航状态系统
- 全量 UI 自动化录制回放

## 风险与回退

风险：

- 若继续把 root-scoped 清理与子树清理共用同一文案而不明确作用域，用户反馈会持续混乱。
- 若多选专注只改按钮可用性、不改内部状态模型，最终仍会退化为单节点专注。
- 若选择恢复、删除、移动和清理后不处理专注集合失效节点，项目树很容易出现隐藏状态残留。

回退方式：

- 优先保持后端 `remove_empty_folders()` 能力不变，把回退范围控制在项目树 UI 状态和专注状态管理层。
- 多选专注若中途发现边界过大，应先保住同一项目内联合专注，再决定是否推迟跨项目策略。
