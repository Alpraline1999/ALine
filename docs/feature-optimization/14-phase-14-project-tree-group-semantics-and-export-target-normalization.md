# Phase 14：项目树分组语义与导出目标归属规范化

## 目标与完成定义

目标：

- 彻底收口“节点属于哪个集合”和“用户在树上看到什么语义”之间的一致性，避免同名文件夹、导出目标、树过滤与页面路由之间继续出现分组漂移。
- 把项目树根分组、普通子文件夹、导出目标创建、目标选择提示和页面 focus/filter 的规则整理成统一契约。

完成定义：

- 数据集、分析结果、数字化、图片集、源文件等根分组与普通子文件夹的语义边界被明确记录并由测试守护。
- 导出/保存流程创建新目录时，不再依赖隐式默认行为决定 `group_type`。
- 树过滤、目标选择器、路径标签、图标展示、页面 focus 路由对同一节点的分组理解保持一致。
- 为后续项目树与导出工作流优化提供统一基线，而不是继续在页面级补丁中分散修补。

## 当前审查结论

本轮 phase12-13 的回归复查确认，当前项目树与导出链路还存在一类未被原方案覆盖的“分组语义漂移”问题：

- 分析页导出结果曲线时，自动创建目标目录的语义依赖共享 helper 默认行为，缺少显式分组约束。
- 项目树 folder icon 规则未严格区分“根分组文件夹”和“普通子文件夹”，导致普通子文件夹错误继承系统分组图标。
- 页面导出目标选择、树过滤与后续右键移动/激活路由虽然大体可用，但对 `group_type`、祖先链和 root label 的理解仍分散在多个 helper 中。

这类问题不是单个页面的偶发 bug，而是“项目树分组契约未完全显式化”的结果。

## 主要问题

### 1. 导出目标创建仍存在隐式分组继承

影响：

- 同名目录在不同分组下的语义容易混淆。
- 后续再扩展批量导出、结果复跑或跨页保存时，容易继续出现“创建到了错误集合”或“显示像错集合”的问题。

### 2. 根分组与普通子文件夹缺少统一展示规则

影响：

- 用户在树上看到的 icon 与真实分组语义可能不一致。
- 继续扩展自定义子目录、批量整理、拖放移动时，视觉反馈会误导用户。

### 3. 分组判断逻辑散落在页面、对话框和项目树 helper 中

影响：

- 规则变更时容易只改一条链路，漏掉其它目标选择、路径标签或页面 focus 分支。
- 测试即使覆盖局部行为，也难以冻结“全链路一致性”。

## 推荐优化方向

### A. 建立显式的“树分组语义契约”

建议统一明确：

- 哪些 folder 是 root group
- 哪些 folder 是普通子文件夹
- 新建/导出创建时何时写入 `group_type`
- 何时允许仅依赖祖先链判断集合归属

### B. 收口导出目标创建与路径标签逻辑

建议把下列能力明确下沉到共享 helper：

- 创建目标目录时的 root group / folder group 决策
- 目标选择器显示 label 与默认 target 的一致性
- 同名目录在不同集合下的区分提示

### C. 为项目树展示与页面路由增加一致性守护

建议补充窄测覆盖：

- root group icon 与普通 folder icon
- 导出目标自动创建后的树归属
- focus tree 过滤下的节点可见性
- 页面路由对 folder / data_file / analysis_result 的目标归属判定

## 验收标准

- 项目树分组语义规则形成单一文档约束。
- 导出/保存创建目录的 group 归属显式化。
- 子文件夹展示规则统一，不再继承根分组 icon。
- 新增窄测覆盖根分组、子文件夹、导出目标创建与页面路由的一致性。

## 本阶段不纳入的范围

- 项目树整体视觉重设计
- 全量树组件自动化 UI 回放
- 分析页/数字化页完整导出流程重做
- 新的批量导出交互

## 风险与回退

风险：

- 若继续依赖隐式默认 `group_type`，后续页面新增导出入口时仍会重复出现同类回归。
- 若只改 icon 不改导出与分组契约，用户面语义仍然会继续漂移。

回退方式：

- 先把共享 helper 与树展示规则显式化。
- 再按窄测补齐页面级和对话框级一致性守护。
