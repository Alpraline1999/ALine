# Phase 10：静态检查、运行时导入与关键工作流守护栏

## 目标与完成定义

目标：

- 把这次已经暴露出来的 `NameError` / 未定义名称 / 重构后旧符号残留问题，收口为一条明确的开发守护栏。
- 避免后续功能优化阶段再次被“代码能启动一部分，但关键入口运行时报未定义名称”这类低层错误打断。

完成定义：

- 仓库存在稳定、低成本的未定义名称静态检查。
- 至少一组关键 UI 导入/启动烟测可以在本地快速执行。
- 项目保存、设置页扩展管理、项目树右键菜单这类高风险入口具备最小自动校验。

## 当前审查结论

本轮已经确认，这不是单一的 `project_manager.py` 漏了 `import re`。

静态检查实际抓到的同类问题包括但不限于：

- `core/project_manager.py`
  - `_safe_filename()` 使用 `re` 但未导入，直接导致保存项目时报错。
- `core/global_assets.py`
  - 重复/导出扩展配置时使用 `copy.deepcopy()`，但未导入 `copy`。
- `ui/pages/settings_page.py`
  - support 模块拆分后，页面中残留了失效的旧常量和类名引用。
- `ui/widgets/project_tree.py`
  - 删除确认对话框使用 `MessageBox`，但未导入。
- `ui/widgets/project_tree_menu_commands.py`
  - 全局资产右键菜单在拆分后保留了未传入的 `pos` 变量和未注入的扩展配置回调。
- `core/extension_types.py`
  - 使用 `Tuple` 注解但未导入。

这说明当前问题的本质是：

- 模块拆分后的接口漂移没有被守护栏及时拦住。
- 当前结构门禁更关注“大方向架构边界”，但对“细粒度运行时名称正确性”的覆盖还不够。

## 主要问题

### 1. 静态未定义名称缺少常驻门禁

影响：

- 很多错误在页面真正执行到某个分支前不会暴露。
- 用户会在保存、右键菜单、设置页等高频入口上直接遇到运行时异常。

### 2. 重构后 support/helper 接口漂移缺少烟测

影响：

- 页面主类与 support/helper 模块之间的重命名、参数变更、回调注入缺失，不容易在代码审查中被肉眼稳定发现。
- “页面能打开，但某个按钮一点就炸”的问题会反复出现。

### 3. 关键工作流没有最小自动回归

影响：

- 当前主窗口启动烟测不足以覆盖：
  - 保存项目
  - 设置页扩展管理入口
  - 项目树全局资产菜单
  - 导入/导出配置类路径

## 推荐优化方向

### A. 建立仓库级未定义名称检查

建议至少收口以下检查：

- `ruff check --select F821`
- 覆盖目录：
  - `core/`
  - `ui/`
  - `app/`
  - `ai/`
  - `models/`
  - `processing/`
  - `extensions/`
  - `scripts/`

目标不是一次性接入大而重的全量 lint，而是先锁住最容易演化为运行时崩溃的错误。

### B. 扩展现有 `scripts/structure_check.py`

建议新增轻量守护项：

- 关键目录的 `F821` 检查封装
- support/helper 模块被主页面引用时的关键符号存在性检查
- 关键页面导入烟测提示

使其从“结构门禁”扩展到“结构 + 关键静态正确性门禁”。

### C. 增加关键 UI 烟测

建议优先覆盖：

- `MainWindow()` 的 `offscreen` 启动
- `SettingsPage()` 导入与扩展页装配
- `ProjectTreeWidget()` 导入与菜单构建
- `project_manager._safe_filename()` / 保存路径整理

目标是低成本、秒级反馈，不做重型 GUI 自动化。

### D. 为高风险工作流增加最小行为测试

建议优先覆盖：

- 保存项目
- 扩展配置复制/导出/设为默认
- 项目树删除确认对话框路径
- 设置页外部扩展目录卡片装配

可优先做无界面或 `offscreen` 级别的窄测试。

## 验收标准

- 仓库 `F821` 检查保持通过。
- 主窗口与关键页面导入烟测可稳定执行。
- 保存项目、扩展配置菜单、设置页扩展卡片不再因漏导入/旧符号残留触发运行时异常。
- 后续 support/helper 模块重命名或回调注入缺失时，能在开发阶段被更早发现。

## 本阶段不纳入的范围

- 全量类型系统重写
- 全仓库严格 `mypy`
- 大规模 UI 自动化测试平台
- 全量回归测试体系建设

## 风险与回退

风险：

- 若一次性引入过多 lint 规则，会把阶段目标从“守住运行时正确性”扩大为“全面规范整改”。
- 若烟测做得过重，会拖慢日常开发节奏。

回退方式：

- 先锁 `F821`、导入烟测和少量关键工作流。
- 先覆盖最容易因重构漂移而出错的模块，再逐步扩大范围。
