# Phase 20：模板编辑器能力对齐与扩展设置卡片样式收口

## 目标与完成定义

目标：

- 修复设置页扩展标签页中“扩展管理”设置卡片标题样式漂移问题，使其与其他设置卡片的字体大小与字重保持一致。
- 将数据管理页中的 Pipelines 模板、图表模板、报告模板管理能力提升到与扩展配置相近的水平，允许用户在右侧编辑区直接查看、修改并保存模板。

完成定义：

- 设置页中，内置扩展和外部扩展区域的“扩展管理”卡片标题样式与其他设置卡片一致，不再出现字号和字重偏差。
- 数据管理页中，Pipelines 模板、图表模板、报告模板都具备“左侧选择 + 右侧编辑 + 保存/另存”的统一工作流。
- 用户不需要离开当前模板上下文，就能在右侧编辑区完成模板内容修改、校验和保存。
- 内置模板、默认模板、用户模板之间的可编辑边界清晰，不会出现直接覆盖只读模板、误改默认模板或保存语义混乱的问题。
- 模板编辑能力的状态管理、脏状态提示、保存失败反馈与扩展配置编辑保持同等级别的清晰度。
- 补齐窄范围测试与必要的行为验证，不做大规模回归。

## 当前审查结论

这组问题不是单纯的 UI 微调，而是“设置页一致性 + 全局资产编辑能力不对齐”的组合问题：

- 设置页扩展标签页中，内置扩展和外部扩展的“扩展管理”卡片标题样式已偏离其他设置卡片，说明设置卡片样式契约没有完全统一。
- 数据管理页对扩展配置已经具备较成熟的右侧编辑区工作流，但 Pipelines 模板、图表模板、报告模板仍更接近列表式管理。
- 这种能力不对齐会让用户在同一页面中面对两套完全不同的编辑心智模型：
  - 扩展配置可以直接在右侧修改并保存
  - 其他模板却仍停留在“选择后外跳 / 弹窗 / 弱编辑”式路径
- 用户当前反馈已明确指出：
  - 需求 1 是样式一致性缺陷，应直接收口。
  - 需求 2 极其重要，且属于复杂功能优化，必须谨慎规划，不能只做表面入口补齐。

## 主要问题

### 1. 设置卡片标题样式契约出现漂移

影响：

- 设置页的卡片层级感被削弱，视觉一致性受损。
- 若同类卡片标题样式继续分叉，后续设置页改动会越来越难统一。

### 2. 模板管理能力落后于扩展配置

影响：

- Pipelines 模板、图表模板、报告模板缺少扩展配置那种稳定的“右侧编辑器”工作流。
- 用户无法在模板浏览与模板修改之间形成连续操作链路，模板管理成本偏高。

### 3. 多类模板的数据结构并不一致，不能简单复用单一表单

影响：

- Pipelines 模板、图表模板、报告模板各自的数据结构、校验规则、只读边界和预览需求不同。
- 如果只在右侧塞进一个临时编辑框，极易出现字段映射错误、保存语义不明确、默认模板误改等问题。

### 4. 模板保存语义与只读语义需要统一重定义

影响：

- 内置模板/默认模板通常不应被直接覆盖，但用户仍需要基于它们修改并另存。
- 若没有明确的保存、另存、创建副本、只读提示契约，编辑器接入后会制造新的数据破坏风险。

### 5. 右侧编辑器接入后会放大状态管理复杂度

影响：

- 页面需要同时处理模板切换、未保存变更、校验失败、保存成功后的选中态同步。
- 如果不把脏状态、切换拦截和保存反馈一起设计，最终会出现“看似能编辑，实际状态错乱”的回归。

## 推荐优化方向

### A. 先收口设置页扩展卡片样式契约

建议要求：

- 复用与其他设置卡片一致的标题字号、字重、间距定义。
- 不单独为内置扩展/外部扩展卡片保留特殊标题样式。
- 若当前卡片样式由局部样式硬编码导致，应优先回收到共享样式层，而不是继续局部修补。

### B. 以“扩展配置编辑器”为参考，统一模板编辑工作流

建议统一目标工作流：

- 左侧资产树/列表负责选择模板。
- 右侧编辑区负责显示模板详情、可编辑字段和保存动作。
- 用户可在当前页完成查看、修改、保存、另存，而不是跳转到分散入口。

建议原则：

- “参考扩展配置”指的是工作流与状态契约对齐，不要求强行共用完全相同的表单实现。
- 各类模板可共享外层壳层，但内部编辑器应允许按模板类型差异化实现。

### C. 分开定义三类模板的编辑边界

建议至少明确：

- Pipelines 模板：
  - 是否允许直接编辑步骤列表、步骤参数、展示名称与描述
  - 保存后如何与处理页模板应用链路保持一致
- 图表模板：
  - 需要区分图表模板、绘图样式、主题类资产之间的边界
  - 避免用户在右侧编辑器里面对含义重叠或命名冲突的模型
- 报告模板：
  - 需要定义模板正文、名称、可能的元信息字段的编辑与保存规则
  - 保持与分析页导出链路兼容

### D. 统一模板的保存/另存/只读策略

建议统一最小契约：

- 用户模板：
  - 支持直接保存
- 内置模板或默认模板：
  - 默认只读
  - 支持“创建副本”或“另存为”后再编辑保存
- 当模板存在未保存修改时：
  - 切换模板前必须有明确提示
  - 提示语义应与扩展配置编辑保持一致或尽量接近

### E. 为右侧编辑器建立稳定的状态管理边界

建议至少覆盖：

- 当前选中模板标识
- 当前编辑草稿状态
- 是否有未保存修改
- 保存成功后左侧名称/列表同步
- 保存失败后的错误反馈与回滚策略

建议要求：

- 不要把三类模板的状态散落在多个临时变量中。
- 若现有数据管理页已经存在扩展配置右侧编辑状态容器，应优先抽象共享壳层，而不是为每类模板再复制一套近平行逻辑。

### F. 以窄范围测试守住复杂编辑契约

建议优先补充：

- 设置页“扩展管理”卡片标题样式一致性的 UI 测试或可替代的结构性验证
- 数据管理页中三类模板右侧编辑器可加载当前模板内容的测试
- 用户模板保存成功后的数据更新测试
- 内置/默认模板不可直接覆盖、只能另存的测试
- 模板切换时未保存变更提示的测试

## 验收标准

- 设置页扩展标签页中的“扩展管理”卡片标题样式与其他设置卡片完全一致。
- 数据管理页中的 Pipelines 模板、图表模板、报告模板均可在右侧编辑区直接修改。
- 用户模板支持保存，内置/默认模板具备只读与另存策略，不会被直接覆盖。
- 模板切换、保存、另存、失败提示和未保存变更提示形成清晰、稳定的编辑工作流。
- 相关改动具备窄范围测试或等价验证，能守住模板编辑契约，不依赖全量回归。

## 本阶段不纳入的范围

- 全局资产系统的全面重构
- 模板协议与存储格式的大规模迁移
- 处理页、可视化页、分析页全部模板应用链路的统一重写
- AI 模板或 AI 配置体系的并行重做

## 风险与回退

风险：

- 若把三类模板强行塞进同一份表单实现，而不考虑字段模型差异，保存链路很容易出现错写或漏写。
- 若只补 UI 入口，不同步保存/另存/只读语义，右侧编辑器会直接放大数据破坏风险。
- 若不复用扩展配置编辑器已有的状态管理经验，DataPage 很可能继续堆积平行逻辑并再次失控。
- 若图表模板的资产边界没有先厘清，实施时容易把图表模板、绘图样式、主题类资源混在一起，进一步放大模型歧义。

回退方式：

- 优先将回退范围限制在数据管理页模板编辑器入口和模板保存路径，不回退底层模板数据本体。
- 若阶段实施中发现三类模板无法一次性完全对齐，应先保证统一编辑壳层与只读/另存契约成立，再逐类补齐深层编辑能力。
